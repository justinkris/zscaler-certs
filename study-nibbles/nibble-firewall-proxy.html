STUDY NIBBLE

Firewall Module vs Proxy Module
Inside the ZIA Public Service Edge

ZIA Architecture Access Control · 10%
THE CORE IDEA

Two modules, one enforcement node

Every ZIA Public Service Edge (formerly Zscaler Enforcement Node) logically splits traffic processing into two modules: the Firewall Module and the Proxy Module. The split exists so the right depth of inspection is applied to the right traffic — without sending every packet through the full security stack.

Key point: Proxy Module = web traffic (HTTP, HTTPS, FTP) with full deep inspection. Firewall Module = everything else (all ports, all protocols) with L3/L4 control plus DPI, DNS, and IPS engines.
SIDE BY SIDE

Firewall Module vs Proxy Module

Dimension Firewall Module Proxy Module
Traffic scope All ports, all protocols (TCP, UDP, ICMP, non-web) HTTP, HTTPS, and some ALGs (e.g. FTP)
Inspection depth L3 / L4 flow control with per-packet L7 identification Full content inspection — headers, payload, HTML, files
SSL inspection Not performed here — this module doesn't decrypt TLS Yes — decrypt, inspect, re-encrypt
Core engines DPI · DNS · IPS · Policy URL Filter · Cloud App Control · ATP · Malware · DLP · File Type · Bandwidth · Browser Control
Analogy Triage nurse — checks everyone at the door, fast Specialist doctor — full examination of specific patients
FIREWALL MODULE — 4 ENGINES

What lives inside the Firewall Module

DPI Engine
Identifies the application from the first 1–5 packets. Catches apps hiding on non-standard ports (e.g. something sneaking on :443 that isn't HTTPS).
DNS Engine
Header-based restrictions and domain-level access control for plain DNS (UDP/TCP 53).
IPS Engine
Per-packet threat detection via signatures — alerts and blocks intrusion attempts across non-web flows.
Policy Engine
Evaluates firewall rules top-down, first-match. Controls what the other three engines do and when.
MENTAL MODEL

The hospital analogy

⟳
ANALOGY
Firewall Module = triage nurse at the hospital entrance. Fast, broad, sees everyone. Asks "who are you, where are you going, are you allowed in this wing?" — identifies the protocol, checks the rule, lets it through or blocks it.

Proxy Module = specialist doctor. Only sees specific patients (web traffic), but performs a full examination: bloodwork, imaging, diagnostics (SSL decrypt, content inspection, DLP, malware analysis).

You wouldn't want the specialist examining every passerby — too slow, overkill. You wouldn't want the triage nurse performing surgery — wrong tools. That's why the modules are separate: right depth of inspection, right traffic, right performance cost.
GOTCHAS

Traps to watch for

!
WATCH OUT — DNS vs DoH
Plain DNS (UDP 53) → Firewall Module → DNS Engine. No SSL inspection possible because it's unencrypted — nothing to decrypt.

DNS over HTTPS (DoH) → rides on HTTPS :443 → Proxy Module. SSL inspection can apply here. This is why DoH is a visibility concern — it hides DNS lookups inside encrypted web traffic, bypassing the DNS Engine unless you inspect it.
!
WATCH OUT — Non-standard ports
If HTTPS runs on a non-standard port (e.g. 8443), the DPI Engine inside the Firewall Module identifies it as web traffic within the first few packets and hands it off to the Proxy Module. Zscaler caches that mapping so future sessions don't need re-discovery.
EXAM ANGLE

How this shows up on the exam

→
EXAM TIP
Expect a scenario question giving you a protocol and port (e.g. "user sends SSH on TCP 22" or "DNS on UDP 53") and asking which module processes it or whether SSL inspection applies. Default test: is it HTTP/HTTPS/FTP? → Proxy. Anything else? → Firewall. SSL inspection only applies where there's a TLS session to decrypt.
→
EXAM TIP
Know the four Firewall Module engines by name: DPI, DNS, IPS, Policy. Know that the Proxy Module is where the "full security stack" runs — URL filtering, Cloud App Control, ATP, malware, DLP, sandbox. If the question names one of those capabilities, the traffic is in the Proxy Module.
