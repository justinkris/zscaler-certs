STUDY NIBBLE

ATP vs IPS

ZIA Cyberthreat Protection · 12% Threat Engines
Both engines inspect inline — they're asking different questions
Traffic flows through Zscaler and hits both ATP and IPS. But they're not doing the same thing. ATP is asking: "Is this file or destination known bad?" IPS is asking: "Is this conversation behaving like an attack?" One looks at what is moving. The other looks at how it's moving. You need both — because a clean file delivered suspiciously is still suspicious.
AT A GLANCE

The two engines, side by side

ATP
Is this thing known bad?
"Check the file, check the URL, check the destination against what we know."
  • Threat intel DB → heuristics → ML → sandbox
  • Catches malware downloads, phishing URLs, C2 domains
  • Covers both outbound requests and inbound file responses
  • Flags based on reputation and content analysis
  • Blind without SSL inspection
IPS
Is this traffic acting like an attack?
"Doesn't matter if the domain is clean — the packet stream looks wrong."
  • Deep packet inspection against signature database
  • Catches XSS payloads, exploit kits, malicious redirects, C2 patterns
  • Primarily inbound response traffic to users
  • Flags based on how traffic behaves, not what it is
  • Blind without SSL inspection
WORKED SCENARIO

The clean file that isn't clean

THE SETUP
A user downloads a file from a reputable domain — but something is wrong with how it arrived
A user at an org visits a site they use regularly. The domain has a clean reputation. The file download passes ATP's threat intel check, clears heuristics, and the ML engine gives it a pass. ATP has no objection — nothing here matches a known bad signature.

But as the response packets arrive, IPS starts watching the pattern. The payload structure matches a known exploit kit delivery profile. There are timing anomalies. Unexpected callbacks embedded in what should be a static file download. The conversation itself looks like an attack.
ATP'S VIEW
Passes ⚠️
Domain reputation: clean. File hash: unknown but no match. Heuristics: marginal. ATP doesn't have enough signal to block — it would need the sandbox to catch this, and sandbox runs asynchronously after delivery.
IPS'S VIEW
Blocked ✓
IPS doesn't care what the file is — it sees the packet stream and matches a known exploit kit delivery signature. The conversation pattern alone is enough to block the session before the payload completes delivery.

WORTH REMEMBERING
★
ONE-LINE MODEL
ATP = the what. IPS = the how. If you can name a bad file, URL, or domain — that's ATP territory. If the scenario describes suspicious behaviour, packet patterns, or exploits in the stream — that's IPS territory. Both require SSL inspection to see inside encrypted traffic.
★
C2 OVERLAP
C2 (command-and-control) traffic sits in both engines' scope — ATP flags known C2 domains by reputation, IPS flags C2 communication patterns by behaviour. The same threat, seen from two different angles. This is intentional — not a conflict.
!
EXAM TRAP — SSL INSPECTION
When the exam asks what happens if SSL inspection is disabled, the answer is the same for both: ATP and IPS both go blind. Don't assume one is more affected than the other. Neither can inspect inside HTTPS traffic without SSL inspection being on.
→
EXAM TIP — SCENARIO ROUTING
When a question describes a scenario and asks which engine is responsible: File download, known bad URL, or C2 domain? → ATP. XSS, exploit kit patterns, malicious redirect, or suspicious traffic behaviour? → IPS. If both are plausible — the question will have a clue in whether it's about reputation or behaviour.
