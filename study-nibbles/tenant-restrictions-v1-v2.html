STUDY NIBBLE

Tenant Restrictions
v1 vs v2

ZIA Access Control
The problem both versions solve
To the network, outlook.office.com is just outlook.office.com. URL filtering can't tell whether a user is signing into the corporate M365 tenant or a personal Hotmail account. Tenant Restrictions closes that gap — Zscaler injects HTTP headers into Microsoft-bound traffic so Microsoft can enforce "which tenant is this user allowed to touch?" policy.
AT A GLANCE

The two models, side by side

VERSION 1
Allowlist on the envelope
"Here's my list of allowed tenants — Microsoft, compare & enforce."
  • Full allowlist lives in the header itself
  • Enforced only at sign-in endpoints
  • Tenant-wide allow/block — no user, group, or app granularity
  • Change allowlist = change Zscaler config
  • Misses anonymous / public resource access
VERSION 2
Policy pointer in a signed token
"Here's my ID — call my HQ in Entra for the current rules."
  • One signed JWT header — points to the enforcing tenant
  • Policy lives in Entra, managed by the Entra admin
  • Enforced at sign-in AND resource endpoints
  • Per-user, per-group, per-app granularity
  • Catches anonymous / public resource access
THE HEADERS

What Zscaler actually injects

TRv1 — allowlist written into the request

GET /common/oauth2/authorize HTTP/1.1 Host: login.microsoftonline.com Restrict-Access-To-Tenants: acmewidgets.onmicrosoft.com,partnerco.onmicrosoft.com Restrict-Access-Context: acmewidgets.onmicrosoft.com # Microsoft checks: "Is the target tenant on the allowlist?" # If yes → allow. If no → block. Simple.

TRv2 — signed pointer to an Entra policy

GET /common/oauth2/authorize HTTP/1.1 Host: login.microsoftonline.com sec-tenant-restriction: eyJ0eXAiOiJKV1QiLCJhbGciOiJSUzI1NiJ9... # JWT payload (decoded) identifies ACME's tenant: # tenant_id: 11111111-1111-1111-1111-111111111111 # signed_by: Zscaler proxy (validated by Microsoft) # Microsoft: "Look up ACME's TRv2 policy in Entra. Enforce that."
WORKED SCENARIO

Alice clicks a public SharePoint link

THE SETUP
ACME employee, public file, foreign tenant — no sign-in required

Alice works at ACME Widgets. She's on her corporate laptop, behind Zscaler with SSL inspection enabled for Microsoft traffic.

A friend at a different company, CompetitorCorp, shares a SharePoint link with her: competitorcorp.sharepoint.com/shared/report.pdf. The file's sharing setting is "anyone with the link" — no sign-in needed.

Alice clicks the link. What happens under each version?

TRv1 path: Alice clicks link → Zscaler injects Restrict-Access-To-Tenants → request arrives at SharePoint (resource endpoint) → SharePoint does NOT enforce v1 headers here → file returned to Alice ✗ TRv2 path: Alice clicks link → Zscaler injects sec-tenant-restriction JWT → SharePoint reads JWT, looks up ACME's Entra policy → CompetitorCorp not in allowlist → access denied ✓
TRv1 OUTCOME
⚠️ Allowed
v1 only enforces at login endpoints. Because no sign-in happens, the v1 header is irrelevant here — Alice downloads the file from CompetitorCorp unchallenged. This is v1's infamous blind spot.
TRv2 OUTCOME
🚫 Blocked
v2 enforces at both login and resource endpoints. SharePoint honours the signed JWT, queries ACME's Entra policy, finds no exception for CompetitorCorp, and refuses the anonymous download.
DIFFERENCES AT A GLANCE

What changes between v1 and v2

Dimension TRv1 TRv2
What it controls Sign-in only Sign-in + resource access
Where policy lives Inside the injected header In Entra ID admin portal
Header type Plaintext tenant allowlist Signed JWT referencing a tenant GUID
Anonymous access protection ❌ None ✅ Yes
Per-user / per-group rules ❌ No — tenant-wide only ✅ Yes
Per-application rules (e.g., only SharePoint) ❌ No ✅ Yes
To change allowed tenants Change Zscaler config Change Entra policy
Microsoft Graph endpoints protected ❌ No ✅ Yes
WORTH REMEMBERING
★
ONE-LINE MODEL
v1 = bouncer at the club door only. v2 = bouncer at the door AND inside every room. When a question mentions anonymous access, public links, or Graph APIs — that's v2 territory, v1 misses it.
!
EXAM TRAP
Tenant Restrictions require SSL inspection. If SSL inspection isn't enabled for M365 traffic, Zscaler can't decrypt the request to inject headers — and the whole mechanism silently breaks. This applies to both v1 and v2.
!
EXAM TRAP
Zscaler injects; Microsoft enforces. Don't pick answers claiming Zscaler decides allow/block for tenant restrictions. Zscaler's job is purely to inject the correct header on Microsoft-bound traffic. The enforcement happens at Microsoft's servers using policy defined in Entra (v2) or parsed from the header (v1).
→
SCENARIO TEST
Whenever you see a Tenant Restrictions question, ask two things: (1) Is this a sign-in or a resource access? (2) Where does the policy live? Those two questions alone will answer almost every TR exam item correctly.
★
GRANULARITY WIN
TRv2 lets admins express rules like "Allow PartnerCo tenant, but only for SharePoint Online, only for users in the 'Project-X' AD group." This per-tenant × per-app × per-user scoping is impossible in v1.
