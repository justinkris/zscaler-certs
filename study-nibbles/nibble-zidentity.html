STUDY NIBBLE · ZERO TRUST AUTOMATION

ZIdentity

Unified identity platform · Authentication · API Client Management

ONE-LINE DEFINITION
ZIdentity is Zscaler's centralised identity platform — the single source of truth for who you are, what you're allowed to do, and how you prove it across all Zscaler products.
🏨
THE ANALOGY
Before ZIdentity, each Zscaler product had its own reception desk, key system, and guest list. ZIdentity is the central reception — one check-in, one keycard, access to whichever floors your booking permits.
THREE IDENTITY TYPES MANAGED
👤
TYPE 01
End Users
Employees accessing apps through ZIA, ZPA, or ZDX
🛡️
TYPE 02
Administrators
People managing Zscaler portals and configuration
🤖
TYPE 03
API Clients
Automated systems and scripts calling OneAPI
KEY FACTS
PROPERTY DETAIL
Position in stack Sits above ZIA, ZPA, and ZDX — all products defer to ZIdentity for auth decisions
API auth method OAuth 2.0 — Client Credentials flow → short-lived Bearer Token (JWT)
API client management Created and managed in the ZIdentity portal — issues Client ID + Client Secret
Token lifecycle Client ID + Secret → Token Endpoint → Bearer Token → expires → re-request
Why unified? Before ZIdentity, each product had separate auth. ZIdentity eliminates fragmented identity silos.
⚠ COMMON EXAM TRAPS
⚠ZIdentity is not just for end users — it governs admins and API clients (machines) equally.
⚠OneAPI uses OAuth 2.0, not a static permanent API key. The Bearer Token is temporary and scoped.
⚠ZIdentity sits above ZIA/ZPA/ZDX — it does not belong to any single product.
⚠Scoped tokens = least privilege for machines, just as access policies = least privilege for users.
FLASH RECALL
What are the three identity types ZIdentity manages? +
End Users (employees), Administrators (portal managers), and API Clients (machines / automated systems calling OneAPI).
Where in the Zscaler stack does ZIdentity sit, and why does that matter? +
Above ZIA, ZPA, and ZDX. It means all products defer to ZIdentity for authentication — no more fragmented identity silos per product.
Why is a short-lived Bearer Token more secure than a permanent API key? +
A Bearer Token is time-limited and scoped — if compromised it expires automatically and only grants access to permitted operations. A permanent key is unlimited and doesn't expire.
