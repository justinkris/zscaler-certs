STUDY NIBBLE

CASB Inline vs API-based

ZIA Data Protection
MENTAL MODEL
⟳
ANALOGY
Inline CASB is like a security guard at the front door of the building — they check everyone walking in and out in real time.

API-based CASB is like a security auditor who has been given the keys to every filing cabinet already inside the building — they walk through periodically and inspect what's already there.

Both are needed. Neither alone is sufficient.

Quick comparison

Dimension Inline CASB API-based CASB
Data state Data in motion Data at rest
Position In the traffic path Out-of-band, via SaaS API
Timing Real-time, pre-delivery Scheduled or on-demand scans
Prerequisite SSL inspection API integration / OAuth with tenant
Blocks uploads? Yes — before data leaves No — data already in tenant

What API-based CASB uniquely catches

Because API-based CASB has read access into a SaaS tenant (like Microsoft 365 or Salesforce), it can detect risks that are invisible to inline inspection:

  • Files shared via "anyone with the link" public URLs
  • Files shared externally to non-corporate domains
  • Overly permissive folder permissions (e.g., "Everyone in the company can edit")
  • Sensitive content sitting in public channels or unprotected locations
  • Misconfigured tenant settings (weak password policies, MFA disabled, etc.)

Remediation — the real power

Because API-based CASB can write back through the API, it can actively fix problems — not just report them:

  • Revoke sharing links
  • Change file or folder permissions
  • Quarantine sensitive files
  • Notify file owners of violations
Key point: None of that is possible inline. Inline CASB only sees traffic in flight — once the data is at rest inside a SaaS tenant, inline is blind to it.
★
MENTAL NOTE
Think of inline as the prevention layer, and API-based as the detection + remediation layer for data that's already landed. A proper CASB deployment uses both side by side.
