Task 2.1 — Configure Forwarding Options
  • ZCC Portal → Administration → Forwarding Profile → Add Forwarding Profile named HandsOnLab
  • Windows Driver: Packet Filter Based · ZIA On-Trusted: Tunnel · Tunnel version: Z-Tunnel 2.0
  • System Proxy: Enforce (all sub-options unchecked) · VPN-Trusted & Off-Trusted: Same as On-Trusted (for both ZIA and ZPA)
  • App Profiles → Windows → Add Windows Policy named HandsOnLab → link HandsOnLab Forwarding Profile → Install Zscaler SSL Certificate: On → Groups: All
  • Log into ZCC as student@<StudentFQDN> → confirm Internet Security ON and Private Access ON → Update Policy
Task 2.2 — Enable SSL Inspection
  • Pre-check (Incognito): https://x.com → cert is from the real server (not Zscaler)
  • ZIA → Policy → Web → Access Control → SSL Inspection → Add SSL Inspection Rule → Enable SSL · Order 1 · Action: Inspect → Save → Activate
  • Post-check (new Incognito): https://x.com → cert is now signed by Zscaler Intermediate Root CA
Task 2.3 — Enable SSL Exemption
  • Pre-check: https://health.com → cert says Zscaler (currently being inspected)
  • Add SSL rule SSL Bypass for Health Care · Order 1 · URL Category: Health · Action: Do Not Inspect → Save → Activate
  • Post-check (new Incognito): https://health.com → cert is from the real server again (bypass working)
Task 2.4 — Verify Zscaler CA Installation
  • On Corp: Client PC → run certmgr → Console Root → Trusted Root Certification Authorities → Certificates → confirm Zscaler Root CA is present and not expired
  • Alternatively: Chrome → chrome://certificate-manager/ → Local Certificates → View imported from Windows → confirm Zscaler cert visible
Task 2.5 — Resolve Certificate Pinning Errors

Certificate pinning means an app hardcodes which CA it trusts. Zscaler's MITM cert won't match — the app refuses to connect. The fix is always a Do Not Inspect SSL bypass rule for that app or domain. Two methods are shown here — one using a Cloud App selector, one using a custom URL category.

  • Signal (Cloud App method): Download and try to install Signal → observe the SSL connection error
  • ZIA → SSL Inspection → Add rule Enable Signal · Order 1 · Cloud Applications: Signal (under Instant Messaging) · Action: Do Not Inspect → Save → Activate
  • Kill Signal in Task Manager → re-run SignalSetup → installs successfully
  • ZIA → Analytics → Web Insights → Logs tab → Timeframe: Current Day → Add Filter: SSL Inspected and Bypassed Transaction → Apply → confirm both inspect and bypass events visible
  • Adobe (Custom URL Category method): Download Adobe Reader → observe the error during installation
  • ZIA → Administration → Resources → URL Categories → Add URL Category → Name: Adobe URL List → paste Adobe-hosted domains (copy from Adobe Product Support page, without *) → Save → Activate
  • ZIA → SSL Inspection → Add rule Enable Adobe · Order 1 · URL Category: Adobe URL List (User-Defined) · Action: Do Not Inspect → Save → Activate
  • Kill Adobe Download Manager in Task Manager → re-run Reader installer → installs successfully
Hints
⚡

Rule order is everything in SSL Inspection. Bypass rules must sit at Order 1 (above the catch-all Inspect rule). The first matching rule wins — if Inspect fires first, the app breaks.

💡

Two bypass methods — know both: Cloud App selector (works when Zscaler categorises the app) vs Custom URL Category (works when you manually specify the domains the app connects to). Adobe's case uses the custom method because its domains are too specific.

🔁

Always close/reopen the browser in a fresh Incognito window after SSL rule changes. Existing TLS sessions are cached — you won't see the new cert without a fresh handshake.

⚠

Kill the broken app in Task Manager before retrying installation. The installer process holds file locks — re-running without killing the old process will fail even after the SSL rule is fixed.

Lab 2 done when: x.com shows Zscaler cert, health.com shows the real cert, Signal and Adobe both install cleanly, and Web Insights shows both inspected and bypassed SSL events.

LAB 6 · ZPA TRAFFIC INSPECTION VIA ZIA

Inspect Private App Traffic with ZIA Enforcement

This is the EDU-230 signature lab. ZPA private app traffic is normally not inspected by ZIA — it tunnels directly from App Connector to the user. This lab enables a ZIA enforcement node in that path, so private app traffic gets the same SSL inspection, malware scanning, and DLP that internet traffic gets. It requires a specific toggle on the App Segment and a Forwarding Control policy change.

💡

📍

🔍

Simulate an attacker doing reconnaissance by interacting with a public-facing decoy. Then investigate the resulting alert in the Deception console using ThreatParse, find your submitted credentials in the event logs, and understand the forensic data captured.

Requirements

  • Task 8.1: Open a browser → visit one of the public decoys: http://wordpress.thezerotrustexchange.com or http://helpdesk.thezerotrustexchange.com

Key Concepts to Be Able to Explain
📘

Why decoys generate high-confidence alerts: Legitimate users have no reason to touch decoys — they're hidden. Any interaction = near-certain indicator of compromise or reconnaissance. Zero false positives, high fidelity.

📘

ThreatParse vs Event Logs: ThreatParse gives the plain-English MITRE-mapped attack summary ("attacker attempted credential harvesting via public-facing login page"). Event Logs give the raw forensic detail — exact POST data, timestamps, headers, submitted credentials.

💡

What the decoy captured: Your username string, the password (hashed or plaintext depending on the decoy config), the HTTP method (POST), the host header, the full request. An attacker wouldn't know any of this was logged.

⚠

If recon post_data user column isn't visible in the Event Logs table, click the chevron at the top-left of the table to open the column selector and add it.

Lab 8 done when: your submitted username appears in the ThreatParse Event Logs for the decoy you visited — your "attack" is on record, MITRE-mapped, with full credential data captured.
