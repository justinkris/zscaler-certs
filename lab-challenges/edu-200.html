Configuring SSL Inspection Policies

ZIA · SSL INSPECTION · BYPASS RULES
ZIA Medium
▾
What You Need to Achieve

Enable SSL inspection for all traffic, then create an exemption that bypasses inspection for the Health URL category. Verify the before-and-after certificate state.

Naming Conventions
RuleNameOrderAction
SSL Inspect all trafficEnable SSL1 (after bypass rule exists)Inspect
Health bypassSSL Bypass for Health Care1 (sits above Enable SSL)Do Not Inspect
Requirements
  • Navigate: ZIA Portal → Policy → Web → Access Control → SSL Inspection
  • Pre-check: visit https://health.com incognito → cert is from the real server (not Zscaler)
  • Create rule Enable SSL → Order 1 → no criteria → Action: Inspect → Save → Activate
  • Post-check: revisit health.com → cert is now issued by Zscaler
  • Create rule SSL Bypass for Health Care → Order 1 → URL Category: Health → Action: Do Not Inspect → Save → Activate
  • Post-bypass check: health.com cert is from the real server again (not Zscaler) — close/reopen browser first
Hints
🔁

Always close and reopen browser (or use a fresh Incognito window) to force a new TLS handshake and see the updated cert.

⚡

Rule order matters: bypass rules must sit above the catch-all Inspect rule. Order 1 bypass + Order 2 inspect-all is the correct setup.

Lab 4 done when: google.com cert says Zscaler; health.com cert says the real server's CA.

Provisioning ZPA Infrastructure

ZPA · APP CONNECTOR GROUP · PROVISIONING KEY · LINUX CLI
ZPA Hard
▾
What You Need to Achieve

Create a provisioning key in ZPA, transfer it to the CentOS App Connector VM, and activate the connector. Then verify it appears in the portal and change the default password.

Naming Conventions
ObjectName / ValueNotes
App Connector GroupOnPremLocation: NYC, NY, USA
Update ScheduleSundays at 00:30Set during group creation
Provisioning KeyOnPremMax reuse: 4
CertificateConnectorSelect during key wizard
Key filename (exact)provision_keyCase-sensitive, NO .txt extension!
Key destination on VM/opt/zscaler/var/Copy with sudo cp
New admin passwordZscaler-123!Change from default zscaler
Requirements — Configuration Flow
  • Log in to ZPA Admin Portal on Corp: Win Svr 2016 (required — WinSCP runs from here)
  • ZPA → Configuration & Control → Private Infrastructure → App Connectors → Add App Connector
  • During wizard: Create new key → select cert Connector → create group OnPrem (NYC, NY, USA, Sundays 00:30) → name key OnPrem, max reuse 4
  • Copy the provisioning key text → paste into Notepad → save as provision_key (use quotes to prevent .txt extension) on Desktop
  • On Corp: ZPA Connector VM — start SSH daemon: sudo systemctl start sshd
  • From Win Svr: use WinSCP (session: Connector, IP 10.0.0.4, port 22) to upload provision_key to /home/admin/
  • Stop SSH daemon after upload: sudo systemctl stop sshd
  • Stop ZPA service: sudo systemctl stop zpa-connector
  • Copy key to Zscaler dir: sudo cp provision_key /opt/zscaler/var/
  • Start and enable connector: sudo systemctl start zpa-connector then sudo systemctl enable zpa-connector
  • Check status after ~10s: sudo systemctl status zpa-connector → should show active (running) with ZPA connection established
  • ZPA portal: confirm App Connector listed under the OnPrem group
  • Change admin password on VM: passwd → old: zscaler → new: Zscaler-123!
Hints
⚠

The filename provision_key is case-sensitive. If Notepad saves it as provision_key.txt, the connector will not enroll.

🔍

After enrollment, run sudo ls /opt/zscaler/var/ — you should see .pem key/cert files proving the connector enrolled.

⏱

Update Status in portal may show blank or "Failure" briefly — wait 2 minutes and refresh. "Success" = already up to date; "Scheduled" = update pending at next interval.

🔒

Stop SSH daemon immediately after the file transfer — never leave SSH running on a production connector.

Lab 6 done when: ZPA portal shows the App Connector with status Connected/Active under group OnPrem, and you can log back in with the new password Zscaler-123!

What You Need to Achieve

Verification Tasks

Protecting Against Data Loss

ZIA SDC · DLP DICTIONARIES · DLP ENGINES · DLP POLICY REVIEW
ZIA Medium
▾
What You Need to Find / Understand
  • ZIA SDC → Administration → Data Loss Prevention → DLP Dictionaries & Engines → view the Credit Cards predefined dictionary — note Confidence Score Threshold, High Confidence Phrases, BIN exclusion options
  • Sort by Type → find custom type Social Security Numbers (US) Regex → review the regex pattern configured
  • DLP Engines tab → view Credit Cards engine — note the Boolean expression: Credit Cards dictionary > 0 matches
  • Policy → Data Loss Prevention → find and view rule PCI High / PCI Medium — review Criteria (DLP Engines, URL Categories, Cloud Apps, File Type, File Size, Users/Groups) and notification/evidence settings
  • Analytics → Data Discovery Report → review Top 10 Users sending sensitive content → click Analyze More → explore categories (Tax, Resume, Real Estate)
  • Analytics → Interactive Reports → Data Loss widget → view Top Violations → click View Logs
Key Concepts to Be Able to Explain
📘

Dictionary vs Engine: A dictionary detects specific content patterns. An engine combines one or more dictionaries using Boolean logic. Policies reference engines, not dictionaries directly.

📘

Custom types: EDM (Exact Data Match), IDM (Indexed Document Match), MIP labels, Regex/Phrases — each has a different detection strength and use case.

Lab 11 done when: you can explain the relationship Dictionary → Engine → Policy Rule and identify what makes the PCI High rule trigger and what action it takes.
