OVERVIEW

How to use this lab challenge guide

Each lab below shows you what needs to be configured — the names, values, ports, and outcomes. The click-by-click steps are collapsed by default: try the task from memory first, then expand STEPS if you get stuck. Lab 1 is left expanded — it just brings the environment up.

You are rolling out Zscaler Client Connector to Windows endpoints to control Internet access, broker access to private apps, and measure digital experience. You will configure ZIdentity, ZIA, ZPA, and ZDX, then validate the end-user experience.

LAB 3
Forwarding Traffic with Client Connector

Build a Forwarding Profile and an App Profile that intercepts on every network type, for both ZIA and ZPA.

Task 3.1 — Configure Forwarding Options

Goal: Create a Forwarding Profile that tunnels everything to ZIA over Z-Tunnel 2.0, and a matching Windows App Profile that pushes the SSL certificate to all users.
→
FORWARDING PROFILE — HandsOnLab
  • Windows Driver: Packet Filter Based
  • ZIA action — On-Trusted Network: Tunnel · Tunnel version: Z-Tunnel 2.0
  • System Proxy: Enforce, all sub-options unchecked
  • VPN-Trusted & Off-Trusted: "Same as On-Trusted Network"
  • ZPA action — On-Trusted: Tunnel · VPN-Trusted & Off-Trusted: Same as On-Trusted
→
WINDOWS APP PROFILE — HandsOnLab
  • Rule Order: 1 · Status: Enable
  • Forwarding Profile: HandsOnLab (the one above)
  • Install Zscaler SSL Certificate: On
  • User Groups: All
!
WHY Z-TUNNEL 2.0?
Z-Tunnel 2.0 is required to intercept all traffic (TCP + UDP, all ports) so the ZIA Firewall and full inspection can apply. Z-Tunnel 1.0 only carries web traffic on 80/443.
Steps
  1. On the Corp: Client PC, open Chrome and login to the ZIdentity landing page as student@<Student FQDN>. Click the Zscaler Client Connector tile.
  2. Navigate to Administration → Forwarding Profile.
  3. Click Add Forwarding Profile:
    1. Name: HandsOnLab
    2. Windows Driver Selection: Packet Filter Based
    3. Forwarding Profile Action for ZIA → On-Trusted Network: Tunnel
    4. Tunnel version: Z-Tunnel 2.0
    5. Expand Configure System Proxy Settings → select Enforce; ensure all sub-options are unchecked.
    6. VPN Trusted Network: tick "Same as On-Trusted Network"
    7. Off Trusted Network: tick "Same as On-Trusted Network"
    8. Forwarding Profile Action for ZPA → On-Trusted Network: Tunnel
    9. VPN-Trusted: "Same as On-Trusted Network" · Off-Trusted: "Same as On-Trusted Network"
    10. Click Save.
  4. Navigate to App Profiles → Windows.
  5. Click Add Windows Policy:
    1. Name: HandsOnLab · Rule Order: 1 · Status: Enable
    2. Forwarding Profile: HandsOnLab
    3. Install Zscaler SSL Certificate: On
    4. User Groups: Select All → Done
    5. Click Save.

LAB 4
Configuring SSL Inspection Policies

The Zscaler Root CA is already trusted on the client (installed by Client Connector via the App Profile setting). You need the inline policy that turns inspection on, plus a privacy carve-out for healthcare.

Task 4.1 — Enable SSL Inspection for All Destinations

Goal: Add a baseline rule to the SSL Inspection policy so every HTTPS connection is inspected.
→
REQUIRED RULE
  • Where: ZIA Admin Portal → Policy → Web → Access Control → SSL Inspection
  • Rule Name: Enable SSL · Rule Order: 1
  • Action: Inspect · Activate after saving
★
VERIFY
Open https://health.com in incognito on the Client PC — the certificate is now signed by Zscaler, not the origin.
Steps
  1. Verify SSL is not currently inspected: on the Corp: Client PC, open Chrome incognito → https://health.com. Click the URL filter icon → Connection is secure → Certificate is valid. The cert is from the origin server.
  2. From the ZIdentity landing page, click Zscaler Internet Access.
  3. Navigate to Policy → Web → Access Control → SSL Inspection.
    1. On the SSL Inspection Policy tab, click + Add SSL Inspection Rule.
    2. Rule Order: 1
    3. Rule Name: Enable SSL
  4. Scroll to ACTION → select Inspect.
  5. Save, then Activate.
  6. Verify: close the browser, re-open in incognito, load https://health.com. The certificate is now signed by Zscaler.

Task 4.2 — Enable an SSL Exemption (Health Care)

Goal: Bypass SSL inspection for healthcare sites to preserve user privacy / meet regulation.
→
REQUIRED RULE
  • Rule Name: SSL Bypass for Health Care · Rule Order: 1 (must sit above the Enable SSL rule)
  • Criteria → URL Categories: Health
  • Action: Do Not Inspect · Activate
★
VERIFY
Refresh https://health.com — the certificate is now the origin's, not Zscaler's. (Refresh or relaunch the browser, or you'll see a cached cert.)
Steps
  1. On the Corp: Client PC, open https://health.com; confirm cert is currently from Zscaler.
  2. In the ZIA Admin Portal, navigate to Policy → Web → Access Control → SSL Inspection.
  3. Click + Add SSL Inspection Rule:
    1. Rule Order: 1
    2. Rule Name: SSL Bypass for Health Care
  4. In Criteria → URL Categories, search for and select Health → Done.
  5. Scroll to ACTION → select Do Not Inspect.
  6. Save, then Activate.
  7. Refresh https://health.com (or close/reopen the browser). The certificate is now the origin's, not Zscaler's.

LAB 6
Provisioning ZPA Infrastructure

The CentOS App Connector VM is pre-built with networking, DNS, NTP, and the App Connector RPM. You need to mint a provisioning key, hand it to the VM, and bring it online.

Task 6.1 — Provision an App Connector (mint the key)

Goal: Use the ZPA Admin Portal "Add App Connector" wizard to create a new App Connector Group and a reusable provisioning key, then save the key to a file on the Win Svr 2016 desktop.
→
REQUIRED CONFIGURATION
  • Where: ZPA Admin Portal opened on the Win Svr 2016 → Configuration & Control → Private Infrastructure → App Connectors
  • Provisioning Key option: Create a new provisioning key
  • Certificate: Connector
  • App Connector Group: name OnPrem · Enabled · IPv4/IPv6 DNS unchanged · Software Update Sundays at 00:30 · Location NYC, NY, USA
  • Provisioning Key: name OnPrem · Maximum Reuse 4
  • Save key as: file provision_key (no extension — wrap the filename in quotes when saving from Notepad) on the Win Svr Desktop
!
FILENAME IS CRITICAL
The file must be named exactly provision_key — case-sensitive, no .txt. The connector activation script reads that exact name.
Steps
  1. Login to Corp: Win Svr 2016 as PATRAINING\Administrator / Admin-123!.
  2. Open Chrome → login to Zscaler Private Access (must be done from the Win Svr — WinSCP runs from here later).
  3. Navigate to Configuration & Control → Private Infrastructure → App Connectors.
  4. Click Add App Connector (top-right) and walk the wizard:
    1. Select Create a new provisioning key → Next.
    2. Select certificate Connector → Next.
    3. Click Add App Connector Group → name OnPrem · Enabled · IPv4/IPv6 DNS unchanged · Software Update Schedule = Sundays 00:30 · Location = NYC, NY, USA → Next.
    4. Provisioning Key: name OnPrem · Max Reuse 4 → Next.
    5. Review settings → Save → Done.
  5. Verify on App Connector Groups tab — group exists.
  6. Verify on App Connector Provisioning Keys tab — key exists with max reuse 4.
  7. Copy the key to a file:
    1. Select App Connector Provisioning Keys → click the Copy icon.
    2. Open Notepad → paste the key.
    3. Save to Desktop as "provision_key" (with quotes — prevents Notepad adding .txt) → close Notepad.

Task 6.2 — Activate the App Connector (load the key)

Goal: Move provision_key onto the App Connector VM, swap it into Zscaler's expected directory, restart the connector service, and confirm the connector appears as Updated in the portal.
→
REQUIRED ACTIONS
  • Login to Corp: ZPA Connector with admin / zscaler
  • Temporarily start sshd (then stop it once the file is transferred — security hygiene)
  • Use WinSCP from the Win Svr to upload provision_key to /home/admin/
  • Stop zpa-connector, copy provision_key to /opt/zscaler/var/, start zpa-connector again
  • Verify in ZPA Admin Portal: connector listed, Update Status becomes Success or Scheduled
  • Change the Connector admin password from zscaler to Zscaler-123!
Steps
  1. Login to Corp: ZPA Connector with admin / zscaler.
  2. Start the SSH daemon: sudo systemctl start sshd (password zscaler).
  3. Verify with sudo systemctl status sshd — should be active (running).
  4. On Corp: Win Svr 2016, transfer the file:
    1. Open WinSCP.
    2. Load saved session Connector (IP 10.0.0.4, Port 22) → Login.
    3. Login with zscaler, accept the certificate.
    4. Left panel: navigate to Desktop → select provision_key.
    5. Right-click → Upload → target path /home/admin/ → OK.
    6. Verify upload, close WinSCP.
  5. Stop the SSH daemon: sudo systemctl stop sshd (verify with status).
  6. Activate the connector:
    1. pwd → confirm /home/admin.
    2. ls → confirm provision_key present.
    3. sudo systemctl stop zpa-connector
    4. sudo cp provision_key /opt/zscaler/var/
    5. sudo ls /opt/zscaler/var/ → confirm file copied.
    6. sudo systemctl start zpa-connector
    7. sudo systemctl enable zpa-connector (auto-start on boot).
  7. Wait ~10 s, then sudo systemctl status zpa-connector — confirm active and connected. sudo ls /opt/zscaler/var should now show .pem key/cert files.
  8. On the Win Svr, in ZPA Admin Portal → Configuration & Control → Private Infrastructure → App Connectors: confirm the connector is listed.
  9. Check Update Status:
    • Success = at the latest version.
    • Scheduled = update queued for next interval.
    Optionally click Update Now to trigger immediately.
  10. Click the connector name to expand details.
  11. On the ZPA Connector VM, change admin password:
    1. passwd · current = zscaler · new = Zscaler-123!
    2. Re-enter; exit; log back in with the new password to confirm; exit.
