OVERVIEW

The ZIdentity Airport

This memory palace maps all six Identity Services concepts onto a single international airport terminal. Walk through it in order — each room locks in one key idea. The theme works because identity is all about who you are and how far you're allowed to go.

⚡ RAPID RECALL WALKTHROUGH

Check-in desk → SAML flow, IdP challenge, golden assertion, wristband token. → Upgrade kiosk at VIP gate → Step-up triggers when your AL is too low for the resource. → Departure board → AL1 to AL4, hierarchical, higher = stronger assurance. → Three-lane scanner → Password, Email OTP, TOTP, FIDO — MFA required by default. → Two lounge doors → SAML door (4 IdPs), OIDC door (5 IdPs), both feed ZIdentity. → Blocked side door → Identity Proxy forces all 8 SaaS apps through Zscaler first.

ROOM 1

The Check-In Desk

01
✈ GATE 01
The Check-In Desk
SAML Authentication Workflow
🛂
🎬
THE SCENE
You walk up to the check-in desk — your app request. The agent doesn't recognise you and sends you to a back office: the ZIA desk for public apps, the ZPA desk for private. A fax fires off a SAML Request to a giant floating judge in the clouds (your IdP). The judge challenges you, then stamps a golden wax-sealed envelope (SAML Assertion) and hands it back through your browser. Zscaler rips it open, checks the wax seal — and if legit, slaps a wristband on you (auth token for ZCC) or stamps your passport (cookie for browser). You're through.
→
THE MEANING
Request → Redirect to ZIA (public) or ZPA (private) → SAML Request to IdP → IdP challenges user → SAML Assertion signed & delivered via browser → Zscaler validates digital signature → Issues token (ZCC) or cookie (browser) → Access resumes via Zero Trust Exchange.
!
EXAM TRAP
Token vs Cookie: ZCC clients receive an auth token. Browser-only users receive a cookie. These are not interchangeable — the exam will test this distinction.

ROOM 2

The Boarding Gate Upgrade Booth

02
✈ GATE 02
The Boarding Gate Upgrade Booth
Step-Up Authentication
⬆️
🎬
THE SCENE
You hold a basic economy boarding pass (password only = AL1). You walk toward the VIP lounge (sensitive resource — finance app, critical ZPA app). A bouncer steps out, looks at your pass, shakes his head, and points to an upgrade kiosk. The kiosk demands your phone as a second check (MFA via ZCC). You tap your phone. A gold boarding pass prints out (higher AL). The bouncer nods and waves you in.
→
THE MEANING
User logs in with low AL → Attempts sensitive resource (defined in ZIA or ZPA policy) → ZIdentity detects insufficient authentication level → Step-up triggered → MFA prompt via ZCC → Higher AL satisfied → Access granted.
!
EXAM TRAP
The MFA prompt during step-up is delivered via Zscaler Client Connector — not the browser alone. It's the ZIA or ZPA policy that defines which resource requires which AL, not the user's role directly.

ROOM 3

The Departure Board

03
✈ GATE 03
The Departure Board
Authentication Levels — AL1 to AL4
📋
🎬
THE SCENE
Above the departure hall hangs a giant board — but instead of destinations, it shows four rows. AL1: slow economy bus outside the building. AL2: normal terminal walk. AL3: fast-track express lane. AL4: private jet tarmac with a fingerprint scanner and two armed guards. Each level is visibly harder to reach — you can't skip straight to the private jet without passing through the earlier rows.
→
THE MEANING
AL1 — Password only. Lowest assurance.
AL2 — Password + lightweight second factor.
AL3 — Stronger MFA. Elevated assurance.
AL4 — Strongest MFA. Critical resources only.

Levels are hierarchical — a higher AL includes all lower ones. Policy in ZIA/ZPA defines which resource requires which AL.

ROOM 4

The MFA Security Scanner

04
✈ GATE 04
The MFA Security Scanner
Authentication Methods
🔐
🎬
THE SCENE
The security checkpoint has three lanes and a VIP gate. Lane 1: you type a password on a keypad. Lane 2: you check your email and read a code aloud to the guard (Email OTP). Lane 3: you hold up a spinning clock app on your phone (TOTP). At the VIP gate on the side: press your thumb to a glowing pad (FIDO). There is no bypass exit — the doors only open forward. MFA is mandatory.
→
THE MEANING
Password — single factor, lowest assurance.
Email OTP — one-time code sent to email.
TOTP — time-based OTP via authenticator app.
FIDO — Fast Identity Online, biometric or hardware key.

MFA is required by default in ZIdentity — it is not optional.
!
EXAM TRAP
Questions implying MFA is optional in ZIdentity are traps. It is enforced by default. Any answer suggesting users can authenticate with password only in a default ZIdentity setup is wrong.

ROOM 5

The Two Lounge Doors

05
✈ GATE 05
The Two Lounge Doors
External IdPs — SAML vs OIDC
🚪
🎬
THE SCENE
Two large lounge doors stand side by side. The SAML Lounge door (blue) has four brand stickers: Okta, Entra ID, AD FS, PingFederate — established enterprise names. The OIDC Lounge door (purple) has five stickers: Entra ID, Okta (OIN or Custom), PingOne, Auth0, OneLogin — slightly more modern. Both doors open into the same ZIdentity terminal desk behind them.
→
THE MEANING
SAML IdPs (4): Okta · Microsoft Entra ID · Microsoft AD FS · PingFederate

OIDC IdPs (5): Microsoft Entra ID · Okta (OIN App or Custom) · PingOne · Auth0 · OneLogin

Both are configured in the ZIdentity Admin portal. ZIdentity supports both SAML and OIDC configurations.
!
EXAM TRAP
AD FS is SAML only — it does not appear in the OIDC list. Don't assume all Microsoft products support both protocols. Entra ID supports both; AD FS supports SAML only.

ROOM 6

The Blocked Side Door

06
✈ GATE 06
The Blocked Side Door
Identity Proxy — Zscaler as IdP
🛑
🎬
THE SCENE
A user tries to sneak into the Salesforce VIP section through a side door — bypassing Zscaler entirely. A giant cartoon hand slams the door shut. Big red sign: "ALL TRAFFIC THROUGH ZSCALER." The correct path has the user going through a Zscaler turnstile first. Eight app logos line the wall beyond the gate: Box, GitHub, Google Apps, Microsoft 365, Salesforce, ServiceNow, ShareFile, Slack. The side door doesn't warn you — it just won't open.
→
THE MEANING
Zscaler can be configured as the Identity Provider (IdP) for these 8 SaaS apps:

Box · GitHub · Google Apps · Microsoft Office 365 · Salesforce · ServiceNow · ShareFile · Slack

Any attempt to access these apps without routing through Zscaler fails entirely — not downgraded, not redirected. Blocked.
!
EXAM TRAP
Identity Proxy = forced routing. Accessing a proxy-protected SaaS app without Zscaler doesn't degrade the experience — it fails entirely. Also know the exact list of 8 apps — the exam may present plausible alternatives.

QUICK REFERENCE
SAML ASSERTION
The golden envelope. Signed by IdP, validated by Zscaler via digital signature.
TOKEN vs COOKIE
ZCC = auth token. Browser-only user = cookie. Never mix these up.
STEP-UP AUTH
Triggered when current AL is insufficient for the resource. MFA prompt delivered via ZCC.
AUTH LEVELS AL1–AL4
Hierarchical. Higher AL = stronger assurance. ZIA/ZPA policy sets the requirement per resource.
IDENTITY PROXY
Zscaler as IdP for 8 SaaS apps. Bypassing = blocked entirely, not degraded.

Box · GitHub · Google Apps · Microsoft Office 365 · Salesforce · ServiceNow · ShareFile · Slack
