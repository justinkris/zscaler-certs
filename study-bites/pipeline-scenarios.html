Three transactions — three very different stopping points in the ZIA policy pipeline. Read each column in order and compare horizontally. Watch how the same pipeline produces wildly different outcomes depending on which engine matches first, which settings are enabled, and whether the file is known.

SCENARIO 1
Gambling site blocked
URL Filtering catches it at step 3 — everything downstream never runs
BLOCKED EARLY
Setup: Admin has URL Filtering rule blocking "Gambling" category for all users.
GET /live-betting Host: bet-on-everything.com
  1. Firewall → SSL → SSMA active
  2. Custom Malicious URLs: clean ✓
  3. Cloud App Control: no match ✓
  4. URL Filtering: category = Gambling → Block rule matches ⛔
  5. Steps 4–14: never evaluated
WEB INSIGHTS Action: BLOCKED By: URL Filtering Category: Gambling ATP/Sandbox: N/A (skipped)
Lesson Early termination saves the entire downstream pipeline from running. Log shows only URL Filtering as the blocking policy — ATP and malware engines never got a chance to weigh in. Most bad traffic is caught this cheaply.
SCENARIO 2
Cascading inverts the default
CAC allows Facebook, but URL Filtering still blocks because cascading is ON
POLICY INTERACTION
Setup: CAC allows Facebook. URL Filtering blocks facebook.com. "Allow Cascading to URL Filtering" is ENABLED.
GET / Host: www.facebook.com
  1. Firewall → SSL → SSMA active
  2. Custom Malicious URLs: clean ✓
  3. CAC: Facebook → Allow rule matches. Normally skips URL Filtering...
  4. URL Filtering: cascading is ON → still runs → facebook.com Block rule matches ⛔
  5. Steps 4–14: never evaluated
WEB INSIGHTS Action: BLOCKED By: URL Filtering Rule: "Block facebook.com" Note: CAC allowed; cascading forced URL eval
Lesson Default behaviour: if CAC explicitly allows, URL Filtering is skipped. Cascading reverses this — both engines evaluate. Zscaler recommends leaving cascading OFF. Also: if CAC had blocked, that decision sticks regardless of cascading.
SCENARIO 3
Unknown file → Sandbox detonation
All inline checks pass. Sandbox breaks out-of-band and catches it.
SANDBOX CAUGHT
Setup: User downloads installer.msi from a 3-day-old domain. File hash unknown globally. Policy = hold-until-verdict.
# Response: Content-Type: application/x-msi MZ... # Windows exe
  1. GET checks (1–14): all pass — new domain, no signatures ✓
  2. Response: Security Ex, IPS, Page Risk: all pass ⚠
  3. Sandbox hash check: not known (bad or good) ⚠
  4. Malware + File Type: no AV match, MSI allowed ⚠
  5. Sandbox detonation: VM runs file → DLL drop + registry persistence + C2 beacon observed → malicious
  6. ⛔ Delivery blocked (~42s hold). Cloud Effect publishes hash globally.
WEB INSIGHTS Action: BLOCKED By: Cloud Sandbox Verdict: Malicious (behavioural) Time to verdict: ~42s Mode: Hold-until-verdict
Lesson Sandbox is the only engine that breaks from millisecond latency. Inline checks couldn't give a verdict — too novel. Detonation took 42s; user waited. Finding published via Cloud Effect → next customer blocked at reputation without re-detonating.
