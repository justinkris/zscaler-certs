Zscaler Deception

Lures, decoys, and the zero false-positive trap.

CORE PRINCIPLE

Why deception is different

Every other threat control — ATP, IPS, Sandbox — has to judge whether something is malicious. Deception doesn't need to judge. Legitimate users and systems have no reason to ever touch a decoy. Interaction with a lure or decoy is proof of malicious intent by definition.

The rule: Nothing legitimate ever touches a decoy. Therefore, anything that does is an attacker — zero false positives.

Deception is designed for attackers who have already gotten past perimeter controls and are moving laterally inside the network — hunting for credentials, servers, databases, and admin accounts. All the other controls failed. Deception is the net that catches what slips through.

⟳
ANALOGY
Think of a museum that's been broken into. The burglar bypassed the front door alarm. Now they're inside, looking for the vault. Deception is the museum planting fake vault doors throughout the building — any burglar who tries to open one instantly reveals themselves, their location, and their method.
THE ATTACK — STEP BY STEP

From the attacker's point of view

1
Attacker compromises an endpoint A phishing email, a malicious download — somehow they have a foothold on a real user's machine inside the network.
2
Attacker does reconnaissance They scan for IPs, check credential stores, read saved browser passwords, look for network shares — standard lateral movement behaviour.
3
Attacker finds a fake saved credential (a Lure) Zscaler has planted a fake credential on this endpoint. It looks completely real:
\\finance-server-01 · admin · Sup3rS3cure!
4
Attacker tries to connect — DNS resolves the decoy Zscaler has injected DNS records for all fake hostnames. When the attacker's machine queries finance-server-01, DNS returns an address pointing to Zscaler's deception cloud — not a real server.
5
Connection reaches the Decoy in Zscaler's cloud The decoy looks convincingly real — it presents a login prompt, responds like a Windows server, lets the attacker think they're making progress.
6
Zscaler fires an alert — attacker is fully identified Every interaction is recorded. The security team now knows: which endpoint was compromised, which user account, which lure was picked up, what technique was used, and precise timing for incident response.
★
MENTAL NOTE
Lures are planted to be found. Decoys are the destinations lures point to. Lures live on real endpoints. Decoys live in Zscaler's cloud. DNS is the glue that routes attacker traffic from one to the other.

The moment an attacker touches a decoy, Zscaler captures: the compromised endpoint, the associated user account, which lure was used, the attack technique (RDP, SMB, SSH), and precise timing. All from a single interaction — with zero false positives.

ASSET TAXONOMY

Types of lures and decoys

Type What it is Where it lives
Lures Fake credentials, saved passwords, network shortcuts planted on real endpoints Real user machines — the breadcrumbs
Decoys Fake servers, applications, and network resources that lures point to Zscaler deception cloud
AD Decoys Fake Active Directory objects — service accounts, users, groups Customer's Active Directory
Honey Tokens Fake files and documents with embedded beacons that phone home when opened File shares, endpoints
!
WATCH OUT
Deception is a post-breach detection control — not a prevention control. It does not stop the initial compromise. Its value is catching attackers who have already gotten inside and are moving laterally. Don't confuse it with perimeter controls like ATP or IPS.
EXAM

What the exam tests

→
EXAM TIP
If a scenario describes an attacker already inside the network doing lateral movement — the answer is Deception. If it asks why deception has zero false positives — the answer is that no legitimate user or system ever touches a decoy, so any interaction is definitionally malicious.
!
COMMON TRAP
The exam may describe a scenario involving lateral movement and ask which control detects it. Deception is the specific answer — not ATP, not IPS, not Sandbox. Those are perimeter/inline controls. Deception is the post-breach internal detection layer.
★
MENTAL NOTE
Remember the three-part chain: Lure (found on real endpoint) → DNS (Zscaler-injected record routes the traffic) → Decoy (fake server in deception cloud captures the attacker). The chain only works because DNS is under Zscaler's control.

QUICK REFERENCE
LURE
Fake credential or asset planted on a real endpoint. Designed to be found by an attacker doing reconnaissance.
DECOY
Fake server or resource in Zscaler's deception cloud. The destination lures point to. Looks real to the attacker.
AD DECOY
Fake Active Directory object — users, service accounts, groups — planted to attract attackers enumerating AD.
HONEY TOKEN
Fake file with an embedded beacon. Opens normally but silently alerts when accessed.
DNS INTEGRATION
Zscaler injects DNS records for fake hostnames so attacker queries resolve to the deception cloud, not real servers.
MY NOTES
Deception = post-breach. If the scenario involves lateral movement inside the network — this is the control, not ATP or IPS.
Zero false positives because the decoys are invisible to legitimate users — only an attacker doing recon will find and use them.
Lure → DNS → Decoy. The three-step chain. DNS is what makes the deception cloud reachable from the attacker's compromised machine.
