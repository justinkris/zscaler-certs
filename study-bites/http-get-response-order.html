OVERVIEW

Two phases: SSMA inspection, then ordered enforcement

The ZIA Service Edge handles every transaction in two distinct phases. Understanding the distinction is the single most important thing to know about policy enforcement — and it's where most people get confused.

THE TWO-PHASE MODEL
SSMA inspects in parallel. Policy enforces in order.
PHASE 1 — INSPECTION
SSMA
Packets placed in shared memory. All engines inspect simultaneously using dedicated CPUs. One scan, many actions. This is where all the verdicts are generated.
PHASE 2 — ENFORCEMENT
Ordered precedence
After SSMA finishes inspecting, policies are applied in a specific order that depends on the HTTP method. First match blocks → downstream policies are skipped.

The enforcement order is different for GET, POST, and Response. Below are the authoritative sequences for GET and Response from Zscaler documentation (POST will be covered in a separate study bite). ATP's individual checks are spread across both — ATP is not a single step.

THE ENFORCEMENT ORDERS

GET · Response — side by side

10
HTTP GET
Request — user fetches a resource
10 policy checks in order
  1. Custom Malicious URLs Checks requested URL against the org's custom ATP denylist. ATP
  2. Cloud App Control Evaluates SaaS app policy. Takes precedence over URL Filtering by default. ACCESS POLICY
  3. URL Filtering Evaluates URL category rules. Skipped if Cloud App Control already allowed (unless cascading enabled). ACCESS POLICY
  4. Security Exceptions If URL is on allowlist → skips Malware, ATP, and Sandbox downstream. ALLOWLIST
  5. Browser Control Checks browser + plugins against Browser Control policy. BROWSER
  6. Country-Based Blocking Checks if destination server is in a blocked country. ATP
  7. IPS Signature Detection Signature-based checks — botnets, XSS, phishing, malicious active content. ATP
  8. Suspicious Content Protection Calculates Page Risk score. Blocks if score exceeds threshold. ATP
  9. P2P Control Checks for prohibited P2P file sharing, anonymizers, or P2P VoIP apps. ATP
  10. Bandwidth Control Allocates full or reduced bandwidth based on policy and current usage. BANDWIDTH
9
GET RESPONSE
Response — data coming back
9 policy checks in order
  1. Security Exceptions If responding URL is on allowlist → skips Malware, ATP, Sandbox. ALLOWLIST
  2. IPS Signature Detection Signature-based response-side threat detection. ATP
  3. Suspicious Content Protection Page Risk scoring on returned content. ATP
  4. Sandbox — known malicious files Checks response for files previously identified as malicious by Sandbox. SANDBOX
  5. Malware Protection AV scan of files in response. Dedicated threads for large files. MALWARE
  6. File Type Control Checks if response files violate file type policy. FILE TYPE
  7. Sandbox — unknown files Behavioural analysis of files not previously seen. Full detonation. SANDBOX
  8. AI/ML-based Content Categorization Categorises uncategorised sites based on content, then re-applies URL filtering. ML
  9. Bandwidth Control Bandwidth allocation. BANDWIDTH
SSL / CONNECT — THE EARLIER GATE

For HTTPS, policy evaluation runs twice

Stage 1 — CONNECT request or SSL Client Hello (SNI)

Before SSL Inspection decrypts anything, only the destination domain is known — not the full URL. So a subset of policies runs based only on the host header or SNI. These are the only checks that can happen pre-decryption.

POLICIES AVAILABLE AT CONNECT / SNI
  • Known Malicious URLs (ATP)
  • Cloud App Control
  • URL Filtering
  • Bandwidth Control

The SSL Inspection policy itself is evaluated on the CONNECT request. If the destination is exempted → bridge the traffic, no inspection, no downstream policy. If blocked → 403 (explicit proxy) or connection reset (transparent). If allowed and decrypted → the full GET / Response order runs on the decrypted HTTP transaction.

!
IMPORTANT — ADVANCED THREAT BLOCKS ARE HARDER
When the destination is known to contain an advanced threat, the connection terminates immediately — even if SSL Inspection is disabled or EUN-for-blocked is enabled. If the block happens on CONNECT host header → 403 error. If the block happens on SNI → connection reset.
KEY FACTS TO LOCK IN

Patterns across the two flows

Pattern What it tells you
Custom Malicious URLs is always step 1 on GET Admin-defined ATP denylists override everything. It's the first gate after Firewall + SSL.
Cloud App Control fires before URL Filtering If CAC explicitly allows, URL Filtering is skipped by default. Cascading must be enabled to change this.
Security Exceptions always appear after URL Filtering The allowlist only kicks in after access policies. It bypasses Malware, ATP, and Sandbox downstream.
Sandbox only exists on the Response flow Makes sense — there's no file to detonate until data comes back from the server.
IPS Signature Detection fires on both GET and Response Request-side (URL/headers) AND response-side (payload) — IPS is the most pervasive check.
ATP checks are distributed, not clustered ATP isn't "one step." It's multiple separate checks spread across the sequence: Custom URLs, Country-Based, IPS, Page Risk, P2P.
AI/ML Categorization only fires on Response It categorises after seeing content, then loops back into URL Filtering — which is why it can't run upstream.
Bandwidth Control is always last Once everything else has decided "allow," bandwidth determines throttling. It's the final gate.
COMMON MISCONCEPTIONS CLEARED UP
!
MYTH 1 — "ATP is a single step"
Wrong. ATP is a collection of checks: Custom Malicious URLs, Country-Based Blocking, Malicious URL Block, IPS Signature Detection, Suspicious Content Protection (Page Risk), and P2P Control. These are spread throughout the enforcement order — not grouped together.
!
MYTH 2 — "URL Filtering fires before ATP"
Partially wrong. Custom Malicious URLs (ATP) is the first check, before URL Filtering. But the rest of ATP (IPS, Page Risk, P2P) fires after URL Filtering. ATP is distributed across the order.
★
MENTAL MODEL — CAC vs URL Filtering
If a Cloud App Control rule explicitly allows a SaaS app, URL Filtering is skipped for that transaction by default. But if CAC blocks, the block applies regardless. The "Cascading to URL Filtering" setting changes this — allowing URL Filtering to still evaluate even after CAC allowed.
→
EXAM TIP — HOW TO READ SCENARIO QUESTIONS
When a scenario asks "which policy blocks this traffic" — trace the GET or Response list in order. Whichever engine matches a block rule first wins, and everything downstream is skipped.
★
NON-WEB TRAFFIC (SSH, RDP, raw TCP)
Outbound non-web traffic going to ports other than 80/443 is sent directly to the Firewall module — the web module and its entire GET / Response order do not apply. If no firewall policy is enabled at the location, Zscaler neither scans nor applies policy to non-web traffic.
