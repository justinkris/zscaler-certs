OVERVIEW

What happens when a user makes a GET request?

An HTTP GET request is a user asking to retrieve something from the web — a web page, an image, an API response. When that request passes through the ZIA Public Service Edge, it doesn't get evaluated by a single policy. It travels through a specific sequence of enforcement stages, and if any stage decides to block it, the journey stops there.

Understanding this sequence end-to-end is critical for the ZDTE exam. Many scenario questions are built around what happens when two policies conflict, or why a particular policy never fires.

Core principle: A GET request first hits the Firewall Module, then — only if it passes — enters the Web Module. Inside the Web Module, policies are evaluated in a fixed order. The moment any policy blocks the transaction, all subsequent policies are skipped.
PHASE 1 — FIREWALL MODULE

The first gate: Firewall

All outbound web traffic — including HTTP GET requests — is sent to the Firewall Module first. This only applies if your organisation has a firewall policy enabled for the user's location.

FW
Firewall Policy Evaluation
Traffic is evaluated against the organisation's firewall rules. If a firewall policy blocks the request, the transaction ends here. If no firewall policy is violated, traffic proceeds to the Web Module.
FIREWALL
!
WATCH OUT — Firewall wins over Web
If the web policy allows an application (e.g. Box.com via Cloud App Control) but the firewall policy blocks it, the firewall wins. The traffic is blocked before it ever reaches the web module. Firewall always has precedence over web policies.
PHASE 2 — WEB MODULE (HTTP GET ORDER)

Inside the Web Module: GET request pipeline

Once traffic clears the Firewall Module, the ZIA Public Service Edge passes it to the Web Module. For HTTP GET requests, policies are applied in this exact sequence. The pipeline stops at the first violation.

COMPARISON

GET vs POST — key differences

For the exam, you need to know what policies appear in GET but not POST, and vice versa. The key distinctions are around file-level policies and DLP — those only activate when data is being uploaded (POST).

Policy HTTP GET HTTP POST
Custom Malicious URLs ✓ Yes ✓ Yes
Cloud App Control ✓ Yes ✓ Yes
URL Filtering ✓ Yes ✓ Yes
Security Exceptions ✓ Yes ✓ Yes
Browser Control ✓ Yes ✓ Yes
Country-Based Blocking ✓ Yes ✓ Yes
Malicious URL Block (global DB) ✗ No ✓ Yes
IPS Signature Detection ✓ Yes ✓ Yes
Suspicious Content Protection ✓ Yes ✓ Yes
Malware Protection ✗ No ✓ Yes
File Type Control ✗ No ✓ Yes
Data Loss Prevention (DLP) ✗ No ✓ Yes
P2P Control ✓ Yes ✓ Yes
Bandwidth Control ✓ Yes ✓ Yes
The pattern: GET is about reading — so file upload, DLP, and Malware Protection on outgoing content don't apply. POST is about sending data — those policies kick in to protect against data leakage and malicious uploads.
MENTAL NOTES

Things to remember

★
MENTAL NOTE — First-block wins
The moment any policy blocks a GET request, the pipeline stops. All remaining policies are skipped. This is why policy order matters — a block on step 2 means steps 3–10 never execute.
★
MENTAL NOTE — Cloud App Control short-circuits URL Filtering
If Cloud App Control explicitly allows an app, URL Filtering is skipped by default. You need to enable "Allow Cascading to URL Filtering" in Advanced Settings to override this. But if Cloud App Control blocks the app, URL Filtering never gets a say — it's already dead.
★
MENTAL NOTE — Security Exceptions is your fast-pass
If a URL is on the Security Exceptions allowlist, it skips Malware Protection, all ATP policies, and Sandbox. Think of it as a VIP bypass lane — once you're on the list, the security checks from step 6 onwards don't apply to you.
⟳
ANALOGY — Airport security
Think of a GET request passing through an airport. The Firewall Module is passport control — if you're on the no-fly list, you never even reach the gate. The Web Module is the security screening lane, with each policy being a different checkpoint (bag scanner, metal detector, explosive swab). The first checkpoint that flags you ends your journey. Security Exceptions is TSA PreCheck — you skip most of the line.
EXAM TIPS

What the exam tests

→
EXAM TIP — Conflicting policy scenarios
Classic exam question: "Cloud App Control allows Facebook, but URL Filtering blocks facebook.com. What happens?" — Answer: Facebook is allowed, because Cloud App Control runs before URL Filtering, and by default, an explicit Cloud App Control allow skips URL Filtering.
→
EXAM TIP — Firewall vs Web logging when blocked
If traffic is blocked by the Firewall Module, both Firewall Insights and Web Insights logs show it as blocked. If it passes the Firewall but is blocked by a Web policy, Firewall logs show allowed while Web logs show blocked. Know the logging asymmetry.
→
EXAM TIP — What GET does NOT have
GET requests do not trigger DLP, Malware Protection, or File Type Control. Those only appear in POST. If an exam scenario mentions a user trying to view a page (not upload/send), and asks which policies could trigger — eliminate DLP, Malware, and File Type Control from your answer.
  • Firewall is evaluated before the Web Module for all outbound web traffic
  • Cloud App Control runs at step 2 — before URL Filtering (step 3)
  • Security Exceptions at step 4 bypasses all ATP and Sandbox policies that follow
  • Bandwidth Control is always last — it throttles, it does not block
  • Wildcard URL rules lose to specific matches — abc.example.com beats *.example.com
  • Cascading to URL Filtering must be explicitly enabled — it is off by default

QUICK REFERENCE
SSMA
Single Scan Multi-Action. All inspection engines run in parallel on the same packets — not chained sequentially. Enables low-latency policy enforcement.
ZIA PUBLIC SERVICE EDGE
Zscaler's inline proxy that applies both firewall and web policies. Has two modules: Firewall Module and Web Module.
CASCADING
"Allow Cascading to URL Filtering" — must be explicitly enabled in Advanced Settings. Without it, a Cloud App Control allow skips URL Filtering entirely.
SECURITY EXCEPTIONS
URLs on the allowlist (in Malware Protection or ATP settings) skip all downstream Malware, ATP, and Sandbox checks.
MEMORY HOOK
"C-CAP, URL, SEC, BROWSER, COUNTRY, IPS, SUSPECT, P2P, BW" — Custom Malicious → Cloud App → URL → Security Exceptions → Browser → Country → IPS → Suspicious Content → P2P → Bandwidth.
