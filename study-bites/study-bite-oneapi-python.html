OVERVIEW

What problem does this script solve?

In Labs 15 and 16.5 you did this entirely through Postman — clicking through a GUI to build HTTP requests. The Python script replaces those same clicks with code. Same calls, same auth flow, same JSON bodies — just automated and repeatable.

Core idea: ZIdentity issues you a short-lived Bearer token. Every API call includes that token in its Authorization header. If the token is valid, the call goes through. That's the entire security model.

OAUTH 2.0 FLOW

How authentication works

The script uses the Client Credentials OAuth 2.0 grant type. This is the correct grant for machine-to-machine API calls — there is no user login, no browser redirect, no consent screen. Your application proves its identity directly using a pre-issued Client ID and Secret.

The three parties
🏛 Authorization Server

ZIdentity — issues the access token after verifying your Client ID and Secret. Token endpoint: <zidentity_url>/oauth2/v1/token

🗄 Resource Servers

ZPA OneAPI and ZIA OneAPI — they hold the actual data and accept API calls. They trust tokens issued by ZIdentity.

💻 Client (your script)

The Python script. It has a Client ID (like a username) and a Client Secret (like a password) — created in ZIdentity → Integration → API Clients.

🎫 Access Token

A short-lived JWT (90 minutes by default). Included in every API call as Authorization: Bearer <token>. If it expires, request a new one.

The token request — annotated
requests.post(
📍
POST request — token endpoint only accepts POST, never GET
  token_url,
🔗
Constructed as: zidentity_base_url + /oauth2/v1/token
  data={
📦
data= sends the body as form-encoded — NOT JSON. This is required by the OAuth spec for token requests.
    "grant_type": "client_credentials",
🔑
Tells ZIdentity this is a machine-to-machine auth. No user involved. No redirect URI needed.
    "client_id": CONFIG["client_id"],
🪪
The Client ID auto-generated by ZIdentity when you saved the API Client. Visible in Integration → API Clients.
    "client_secret": CONFIG["client_secret"],
🔒
Shown once only at creation time. If lost, delete and regenerate. Treat like a password — never commit to source control.
  },
  headers={"Content-Type": "application/x-www-form-urlencoded"},
📋
This header tells the server the body is form-encoded, not JSON. Without it, the token request may fail.
)
↩
Response contains access_token, token_type: Bearer, and expires_in (seconds).
How the token is used in every subsequent call
Python 
# The token goes in the Authorization header of every API call
# after the initial token request. This is the same for ALL endpoints.

headers = {
    "Authorization": f"Bearer {token}",
    "Content-Type":  "application/json",  # only needed for POST/PUT
}

# ZIA or ZPA verifies the token with ZIdentity — if valid, the call proceeds
resp = requests.get(url, headers=headers)

API CALLS — ONE BY ONE

What each call does, and why

Five HTTP requests in total. Each one maps to a step in Postman from the lab. Click any card to see the code, the response shape, and where to verify the result in the portal.

POST
Acquire OAuth 2.0 Token
/oauth2/v1/token
ZIdentity
▾
Python — get_access_token() 
token_url = f"{CONFIG['zidentity_base_url']}/oauth2/v1/token"

resp = requests.post(
    token_url,
    data={                                     # form-encoded body — NOT json=
        "grant_type":    "client_credentials",
        "client_id":     CONFIG["client_id"],
        "client_secret": CONFIG["client_secret"],
    },
    headers={"Content-Type": "application/x-www-form-urlencoded"},
)

token = resp.json().get("access_token")   # extract just the token string
Response shape
JSON Response — 200 OK
{
  "access_token": "eyJhbGciOiJSUzI1NiIsIn...",  // the JWT — keep this
  "token_type":   "Bearer",
  "expires_in":   5400                           // 90 minutes in seconds
}

The access_token is a JWT (JSON Web Token). It's a signed string that encodes who you are and what you're allowed to do. You pass it as-is — no decoding needed. ZPA and ZIA verify its signature against ZIdentity's public key.

VERIFY IN PORTAL

ZIdentity → Integration → Tokens — active tokens for your API Client are listed here.

/zcc/papi/public/v1/getDevices
ZCC
▾

url = f"{CONFIG['zidentity_base_url']}/zcc/papi/public/v1/getDevices"

resp = requests.get(
    url,
    headers={"Authorization": f"Bearer {token}"},   # no body for GET

# data is a list of device objects
# each item has: deviceName, userName, osType, zappVersion, lastSeen...
What each device object looks like
JSON — sample device
{
  "deviceName":   "CORP-CLIENT-PC",
  "userName":     "student@zs9901.safemarch.com",
  "osType":       "WINDOWS",
  "zappVersion":  "4.x.x.x",
  "lastSeen":     "2026-04-28T10:22:00Z"
}

A GET request has no body. The only credential is the Bearer token in the header. The ZCC endpoint is hosted on the ZIdentity base URL — not the ZPA or ZIA base URL. This trips people up.

GET
Get Enrollment Certificate ID
/mgmtconfig/v1/admin/customers/:id/enrollmentCert
ZPA
▾
Python — get_signing_cert_id(token) 
customer_id = CONFIG["zpa_customer_id"]
url = (
    f"{CONFIG['zpa_base_url']}"
    f"/mgmtconfig/v1/admin/customers/{customer_id}/enrollmentCert"
)

resp  = requests.get(url, headers={"Authorization": f"Bearer {token}"})
certs = resp.json().get("list", [])
cert_id = certs[0]["id"]                     # this is the signingCertId

This call exists only to get the id value that the next call needs. In Postman you looked at the response body and manually copied the number into the next request. The script does that lookup programmatically — the two functions are chained in main() so the cert ID flows directly into the group creation call.

★
WHY IS THIS NEEDED?
ZPA requires every App Connector Group to reference a signing certificate. The certificate is what the Group uses to authenticate new App Connectors enrolling against it. You can't create the Group without telling ZPA which cert to use.
POST
Create App Connector Group
/mgmtconfig/v1/admin/customers/:id/appConnectorGroup
ZPA
▾
Python — create_app_connector_group(token, signing_cert_id) 
payload = {
    "name":              "ZPALAB APPC Group",
    "description":      "App Connector training Lab at San Jose",
    "upgradeTimeInSecs": "66600",          # scheduled upgrade time
    "latitude":          "37.3382032",     # San Jose, CA
    "longitude":         "-121.886286",
    "location":          "San Jose, CA, USA",
    "signingCertId":     signing_cert_id,  # fetched in the previous call
}

resp = requests.post(
    url,
    json=payload,                            # json= sends body as JSON + sets Content-Type
    headers={"Authorization": f"Bearer {token}"},
)
# success → 201 Created, response body contains the new group's "id"
data= vs json= — a critical Python difference
📋 data= (form-encoded)

Used for the token request only. Sends the body as key=value&key=value pairs. OAuth 2.0 token endpoints require this format by spec.

📦 json= (JSON body)

Used for all ZPA and ZIA API calls. Automatically serialises your dict to JSON and sets Content-Type: application/json. Saves you one header line.

VERIFY IN PORTAL

ZPA → Configuration & Control → Private Infrastructure → App Connector Management → App Connector Groups. The new group appears immediately after a 201 response.

POST
Create URL Filtering Rule — Block Gambling
/api/v1/urlFilteringRules
ZIA · Lab 16.5
▾
Python — create_url_filtering_rule(token) 
payload = {
    "name":           "Block Gambling Sites",
    "order":          1,
    "state":          "ENABLED",
    "action":         "BLOCK",
    "protocols":      ["ANY_RULE"],       # ← critical — see gotcha below
    "urlCategories":  ["GAMBLING"],        # ZIA predefined category
    "requestMethods": ["GET", "POST", "PUT", "DELETE",
                       "HEAD", "OPTIONS", "CONNECT"],
    "blockOverride":   False,
}

resp = requests.post(
    f"{CONFIG['zia_base_url']}/api/v1/urlFilteringRules",
    json=payload,
    headers={"Authorization": f"Bearer {token}"},
)
# success → 200 or 201, response contains the new rule's "id"
# test: browse to https://www.888.com → should be blocked by Zscaler
Why each field exists
"order": 1
📋
ZIA evaluates URL filtering rules top-down. Order 1 = evaluated first. If this matches, lower-priority rules never run.
"state": "ENABLED"
✅
Rules can be created in a disabled state. ENABLED makes it active immediately — same as ticking the Enable checkbox in the portal.
"protocols": ["ANY_RULE"]
⚡
The lab's key gotcha. Without this, the rule is created but won't enforce — it has no protocol scope to match against. ANY_RULE means apply to all protocols (HTTP, HTTPS, FTP…).
"urlCategories": ["GAMBLING"]
🎰
References a ZIA predefined category. The string must match exactly. Find category IDs in ZIA → Administration → URL Categories.
"blockOverride": False
🔒
If True, users could override the block with a justification. False = hard block, no bypass possible.
VERIFY IN PORTAL

ZIA → Policy → URL & Cloud App Control → URL Filtering Policy. The rule appears immediately. Then browse to https://www.888.com — Zscaler should block it.

GOTCHAS & EXAM TRAPS

The things most likely to catch you out

The four things to get right
  • 1
    Token request uses data= not json=. The OAuth spec requires the token request body to be form-encoded (application/x-www-form-urlencoded). If you use json= instead, ZIdentity will return a 400 or 401 with a confusing error. Every other call in this script uses json= — only the token request is different.
  • 2
    "protocols": ["ANY_RULE"] is mandatory for URL filtering rules. The lab guide flags this explicitly. Without it, the API returns a success response (200/201), the rule appears in the portal, but it will never match any traffic — there's no protocol scope to evaluate against. Silent failure — the worst kind.
  • 3
    The Client Secret is shown exactly once. At the moment you click Add in ZIdentity → Integration → API Clients. If you navigate away without copying it, it's gone. You must delete the secret and create a new one. Maximum two secrets per client at any time.
  • 4
    The signingCertId must be fetched — it's not the Customer ID. They're different numbers. The Customer ID identifies your ZPA tenant. The signingCertId is the ID of a specific enrollment certificate. You get it by calling the /enrollmentCert endpoint first, then pass the returned id field into the App Connector Group POST body.
Quick mental model — the whole flow in one view
Pseudocode 
# 1. Prove who you are → get a short-lived pass
token = POST /oauth2/v1/token  {client_id, client_secret, grant_type}

# 2. Use the pass to read data
devices  = GET  /zcc/papi/public/v1/getDevices     {Bearer token}
cert_id  = GET  /zpa/.../enrollmentCert           {Bearer token}

# 3. Use the pass to create things
group    = POST /zpa/.../appConnectorGroup        {Bearer token, json body}
rule     = POST /zia/.../urlFilteringRules        {Bearer token, json body}

# The token is the only thread connecting all five calls
# Expire the token → all calls fail until you request a new one

Token endpoint
<zidentity_url>/oauth2/v1/token — always POST, always form-encoded body.
Grant type
client_credentials — machine-to-machine. No user, no browser, no redirect.
Using the token

data= vs json=

protocols gotcha
