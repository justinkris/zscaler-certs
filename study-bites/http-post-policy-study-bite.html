OVERVIEW

What makes a POST request different?

An HTTP POST request is a user sending data to the web — submitting a form, uploading a file, posting to social media, or sending an email through webmail. Because data is leaving the organisation, the policy chain is significantly longer than a GET request. Zscaler needs to inspect not just the destination, but the content itself.

POST requests share the early pipeline stages with GET (threat checks, app and URL control, browser control), but then diverge: they add Malicious URL checking against the global database, Malware Protection on outgoing files, File Type Control, and most critically — Data Loss Prevention (DLP).

Core principle: Like GET, a POST request first hits the Firewall Module, then the Web Module. But the POST pipeline has 4 additional policies that GET never sees: Malicious URL Block (global DB), Malware Protection, File Type Control, and DLP. These exist because POST carries outbound data that could leak or contain malware.
PHASE 1 — FIREWALL MODULE

The first gate: Firewall

Identical to GET — all outbound web traffic passes through the Firewall Module first. If the organisation has a firewall policy enabled for the location, that policy is evaluated before anything in the Web Module runs. A firewall block here ends the transaction immediately.

FW
Firewall Policy Evaluation
Evaluated against the organisation's firewall rules. A block here prevents the request from ever reaching the Web Module. The firewall always takes precedence over web policies.
FIREWALL
PHASE 2 — WEB MODULE (HTTP POST ORDER)

Inside the Web Module: POST request pipeline

Once traffic clears the Firewall Module, it enters the Web Module. Steps marked POST ONLY do not appear in the GET pipeline — they are unique to POST because data is being sent outbound.

COMPARISON — POST VS GET

Where the pipelines diverge

POST has 14 steps vs GET's 10. The four extra policies all relate to outbound content inspection. Understanding exactly where they sit in the order — and what that means when policies conflict — is prime exam territory.

The pattern: The four POST-only policies — Malicious URL Block, Malware Protection, File Type Control, and DLP — are all about inspecting what is being sent outbound. GET is read-only, so they don't apply. POST sends data, so they're essential.
MENTAL NOTES

Things to remember

★
MENTAL NOTE — File Type Control runs before DLP
File Type Control is step 11. DLP is step 12. This order is critical for exam scenarios. If a user tries to upload a PDF containing Social Security numbers and your File Type Control policy blocks PDFs — the transaction is blocked by File Type Control, and DLP never fires. The auditor notification configured in DLP is never sent. The policy that hits first wins.
★
MENTAL NOTE — Two different URL threat databases
Step 1 (Custom Malicious URLs) checks your organisation's own custom denylist. Step 7 (Malicious URL Block) checks Zscaler's global threat intelligence database — updated daily with partner feeds. They are separate checks. A URL can pass your custom list but still be flagged by the global database.
★
MENTAL NOTE — Malware Protection has a 10 MB thread split
Files larger than 10 MB are processed by dedicated threads so they don't block the evaluation of smaller files. This is a detail Zscaler specifically calls out — expect it to appear in questions about performance or why large file scans don't create a bottleneck.
⟳
ANALOGY — Customs at the departure gate
If GET is arriving passengers going through passport control (who are you, where are you from?), then POST is departing passengers at a customs checkpoint. You not only check who they are and where they're going, but you also open their luggage. Customs inspects what's inside the bag (Malware Protection), checks whether certain items are prohibited to export (File Type Control), and looks for anything sensitive being smuggled out (DLP). The bag inspection only makes sense when something is leaving — hence why these checks don't exist for GET.
EXAM TIPS

What the exam tests

→
EXAM TIP — The File Type Control + DLP ordering trap
Classic scenario: "A user sends a PDF containing US SSNs. File Type Control blocks PDFs. DLP is configured to alert auditors when SSNs are detected. Does the auditor get notified?" — Answer: No. File Type Control (step 11) blocks the transaction before DLP (step 12) ever runs. Once a policy blocks, all subsequent policies are skipped. The DLP notification never triggers.
→
EXAM TIP — Know what POST has that GET does not
If a scenario describes a user uploading, submitting, or sending data — you're dealing with POST. The four policies exclusive to POST are: Malicious URL Block (global DB), Malware Protection, File Type Control, and DLP. These will never be correct answers for a GET-based scenario.
!
WATCH OUT — Custom denylist vs global database
Don't confuse "Custom Malicious URLs" (step 1 — your org's own list) with "Malicious URL Block" (step 7 — Zscaler's global database). They are separate checks at different points in the pipeline. The custom list is always checked first. The global database only appears in POST, not GET.
  • POST pipeline has 14 steps — GET has 10. The 4 extras are all outbound content inspection
  • DLP is the very last content inspection step before P2P and Bandwidth Control
  • File Type Control always runs before DLP — a file type block silences the DLP alert
  • Malware Protection scans outgoing files against AV/anti-spyware signatures
  • Malicious URL Block (step 7) checks the global Zscaler threat DB — POST only
  • Security Exceptions bypass Malware Protection, ATP, and Sandbox — but not DLP or File Type Control
  • Bandwidth Control is always last — same as GET — it throttles, never blocks

QUICK REFERENCE
DLP
Data Loss Prevention. Inspects POST content for sensitive data (PII, SSNs, credit cards). Only runs if all prior steps allow the transaction. POST only.
FILE TYPE CONTROL
Blocks or allows specific file types from being uploaded. Runs at step 11 — before DLP at step 12. A file type block silences any DLP alert.
MALICIOUS URL BLOCK
Step 7 in POST. Checks Zscaler's global, partner-fed threat URL database. Different from step 1 (Custom Malicious URLs), which checks only the org's own list.
MALWARE PROTECTION
Scans outgoing files in POST against AV/anti-spyware signatures. Files >10 MB use dedicated threads. POST only.
MEMORY HOOK
POST = "Custom, Cloud, URL, Sec-Ex, Browser, Country, Mal-URL, IPS, Suspect, Malware, FileType, DLP, P2P, BW" — the four bold items are POST-only. Think of them as the "outbound content inspection block" that only exists when data is leaving.
