OVERVIEW

The two extra stages that only fire on upload

TL;DR
POST traffic follows the same pipeline as GET — up to stage 11 (Suspicious Content). Then two upload-specific engines fire: 11A · Malware Protection (scan the file) and 11B · DLP (scan the data). After that, P2P and Bandwidth run as normal. Malware always runs before DLP — threats outrank policy violations.
THE DIVERGENCE POINT

Up to stage 11, GET and POST are identical. At 11A, the POST pipeline picks up two extra checks before P2P.

01–10 → 11 · Suspicious Content → 11A · Malware 🆕 → 11B · DLP 🆕 → 12 · P2P → 13 · Bandwidth

Why do these only exist for POST?

GET = pulling data in. POST = pushing data out. The two new engines only make sense when data is leaving the organization.

DLP — DATA LOSS PREVENTION
A user downloading a movie isn't losing corporate data. A user uploading a spreadsheet full of customer SSNs is. DLP only has a job to do when the user is sending data out.
MALWARE (POST-SIDE)
Two reasons: (1) a compromised endpoint may be exfiltrating or beaconing home, and (2) an uploaded file could infect others who later download it. POST malware scanning prevents your org from being patient zero or the exfil channel.
THE TWO STAGES

Stage 11A and 11B — deep dive

11A
FILE-LEVEL THREAT SCAN · POST ONLY

🏈 Malware Protection — Pat McAfee

Scans the file being uploaded (executables, documents, archives, macros, scripts) against signature-based and heuristic malware detection. Blocks or quarantines infected files before they reach the destination — whether the destination is a SaaS app, a web form, or an attacker's C2 server.
Back at the Australian Open court — the glowing red risk book from the previous scene suddenly bursts open, and a malicious envelope flies out and starts chasing me. I sprint across the court and hug Pat McAfee, begging for protection.
WHY HERE IN THE ORDER
Malware runs after URL-level checks (IPS, Suspicious Content) because by now the file is actually in hand and ready to inspect. It runs before DLP because security threats outrank policy violations — if a file is malicious, DLP doesn't need to waste cycles scanning it for SSNs; it's already getting blocked.
11B
CONTENT-LEVEL DATA SCAN · POST ONLY

🛡️ DLP — The DATA Shield

Data Loss Prevention inspects the content of the upload against configured dictionaries and engines — credit cards, SSNs, PII, source code, AIP-labeled documents, custom patterns. Can block, notify auditors, or allow with logging. Requires SSL inspection to work on HTTPS traffic.
Pat McAfee pulls out a massive shield labelled "DATA" and uses it to block the envelope mid-air — then PUNTS the envelope away into the stadium crowd. I'm safe. Pat then escorts me through the P2P crowd control and by the end, we're BOTH exhausted — bandwidth completely gone.
Stop-at-first-match: DLP processes uploads in browser order. If file #1 in a multipart upload triggers a DLP rule, file #2 in the same upload is not scanned. Exam trap: "why isn't my DLP alerting on the second file?" → it's because the first one already matched.

EXAM TIPS

Common exam traps

EXAM TIP · DLP REQUIRES SSL INSPECTION
If a scenario says "DLP isn't triggering on uploads to WebA", the #1 diagnosis is SSL inspection is disabled for that domain. DLP can't read ciphertext.
EXAM TIP · FIRST-MATCH STOP
DLP processes multipart uploads in browser order and stops at the first match. Reverse the upload order → different file triggers the alert. This is a common scenario question.
EXAM TIP · EMPTY POST PRECEDING A DATA POST
Some web apps send an empty POST first, then a second POST with the actual data. If a policy blocks the first empty POST, the second one never runs — and DLP never gets to scan the real payload. Check Content-Length in the HTTP header trace.
EXAM TIP · ORDERING PRINCIPLE
Malware before DLP reflects the principle that security threats outrank policy violations. If asked "which policy fires first when a malicious file contains SSNs?" — the answer is Malware Protection (threat), not DLP (policy).
