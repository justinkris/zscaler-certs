THE SCENARIO

When Two On-Ramps Collide

A corporate branch has a GRE tunnel forwarding all internet-bound traffic to Zscaler for ZIA inspection. Endpoints on that same network also have ZCC installed in Tunnel 2.0 mode. Both mechanisms are trying to forward traffic to Zscaler — but they don't know about each other.

What Actually Happens

ZCC Tunnel 2.0 uses DTLS over UDP 443 to establish its tunnel to Zscaler. That DTLS traffic is internet-bound — which means the corporate network's default route catches it and sends it down the GRE tunnel first. You end up with DTLS encapsulated inside GRE.

Zscaler strips the GRE header and sees the DTLS packet — but at this point it has already been processed as ZIA traffic from the branch Location. The ZCC tunnel establishment is now competing with the GRE inspection path.

THE PROBLEMS

What Goes Wrong

TUNNEL 2.0 PROBLEMS
  • DTLS (UDP 443) dropped by corporate firewall or GRE path
  • ZCC falls back to TLS over TCP 443 — loses UDP performance benefits
  • SSL inspection at GRE layer can break ZCC's own TLS tunnel establishment
  • Traffic inspected twice — double latency and policy conflict
TUNNEL 1.0 PROBLEMS
  • Uses HTTP CONNECT — avoids tunnel-in-tunnel breakage
  • But web traffic (80/443) is still forwarded by both ZCC and GRE
  • Double forwarding and double inspection still occur
  • Redundant overhead with no security benefit
!
KEY ISSUE — DOUBLE INSPECTION
When SSL inspection is enabled on the GRE path, Zscaler intercepts and re-signs TLS sessions at the branch location level. ZCC's own encrypted tunnel arrives already inspected and re-signed — which can break ZCC tunnel establishment entirely. The two mechanisms are not designed to stack.
THE SOLUTION

Trusted Network Detection

The correct design is to let ZCC and GRE complement each other, not compete. When an endpoint is on the trusted corporate network, ZCC should detect this and disable its ZIA tunneling — leaving GRE to handle forwarding. ZCC stays active only for ZPA.

Trusted Network Detection (TND)
A ZCC feature that detects when the endpoint is on a known corporate network (via DNS resolution checks, hostname, or other signals) and switches ZCC into No Tunnel mode for ZIA traffic. Eliminates the double-forwarding problem entirely.

Critical rule: The ZPA tunnel from ZCC is always maintained — even in No Tunnel mode on a trusted network. ZPA access is identity-driven, not location-driven. Only the ZIA tunnel steps aside.

Off Trusted Network

When TND detects an unknown network (home, café, airport), ZCC re-enables Tunnel 2.0 and handles both ZIA and ZPA forwarding directly. GRE is not in the picture.

MENTAL NOTES

Things to lock in

★
MENTAL NOTE
DTLS = UDP 443. Most enterprise firewalls and GRE paths drop UDP. ZCC falls back to TLS/TCP automatically — but you lose the performance benefit of DTLS. This is a symptom of misconfigured coexistence, not a feature.
★
MENTAL NOTE
Tunnel 1.0 (HTTP CONNECT) avoids the tunnel-in-tunnel breakage problem but doesn't solve the double-forwarding problem. It's still redundant. TND + No Tunnel is the only clean answer.
⟳
ANALOGY
ZCC and GRE both trying to forward traffic is like two security guards both trying to escort the same person through the same checkpoint. It doesn't make you more secure — it just creates chaos at the gate. TND tells one guard to stand down when the other is already on duty.

QUICK REFERENCE
DTLS
Datagram TLS — ZCC Tunnel 2.0 protocol. Runs over UDP 443. Faster than TLS/TCP but commonly dropped by enterprise firewalls and GRE paths.
TLS FALLBACK
When DTLS is blocked, ZCC automatically falls back to TLS over TCP 443. Works but loses UDP performance benefits.
TRUSTED NETWORK DETECTION
ZCC feature that detects known corporate network and disables ZIA tunneling. GRE handles ZIA forwarding. ZPA tunnel stays up.
NO TUNNEL MODE
ZCC state on trusted network. ZIA traffic forwarded by network (GRE/IPSec). ZCC only maintains ZPA tunnel.
DOUBLE INSPECTION
Traffic hits ZIA policy twice — once at GRE layer, once via ZCC tunnel. Causes latency, policy conflicts, and can break SSL inspection.

CORRECT DESIGN

MY NOTES
ZPA tunnel ALWAYS stays up — even in No Tunnel mode. ZPA is identity-driven not location-driven. Never forget this.
If a question describes DTLS failures or double inspection on a corporate network — the answer is almost certainly misconfigured TND or missing No Tunnel policy.
