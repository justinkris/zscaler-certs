HOW THIS PALACE WORKS

The narrative

Same boy as the GET + POST palace — but as a younger version. This palace maps the HTTP GET or POST Response flow — what happens when data comes back from the server. The 9 scenes cover every policy check Zscaler runs on response content before it reaches the browser.

Echoes from the GET + POST palace

Four scenes deliberately echo imagery from the GET + POST palace: the VIP snake, fans wanting signatures, the glowing SCP book, and the exhausted boy. These policies are checked on both directions of a transaction — same gate, checked again on the response. The new scenes (4–8) are file/content engines that only make sense on the response side: Sandbox, Malware, File Type, and ML categorisation.

The Contract = The File
Policy → File Type Control
NEW
🎨
Upload image here
Young Pat unrolls the contract on a tiny picnic bench beside the sandbox. The paper has a big official seal at the top — golden magic bytes stamped in wax. The contract itself is the file being inspected.

7
The Unknown Contract — Confused Boy
Policy → Sandbox — unknown files (detonation)
NEW
🎨
Upload image here
Pat hands the boy a different, unfamiliar contract he's never seen before. The younger boy looks up, completely confused, still sitting in the sandbox holding the unknown paper. He doesn't know what to do with it — so the sandbox has to figure it out on his behalf.
HOOK — NEW
Unknown contract in the sandbox = Sandbox detonation for unknown files. If the file hash isn't known, it's sent for full behavioural detonation in an isolated VM. This is the slow out-of-band path (seconds to minutes) — the boy's confusion = the file being analysed before a verdict is made.
