THE TRAP QUESTION
User-to-app does not stop lateral movement between devices. It only controls which business applications a user can reach through Zscaler. Ransomware spreading device-to-device on the same LAN bypasses Zscaler entirely — the traffic stays local. Only user-to-user (device-to-device) segmentation, enforced at the endpoint by the Client Connector, can stop this. When you see "ransomware at a branch" or "lateral movement on the same network" → answer is user-to-user.

Private Service Edge — Data Plane vs Control Plane ⚠ GAP

You understood the concept and the compliance driver — but initially answered that the user's web traffic passes through the Zscaler public cloud. This is the opposite of what PSEs are designed to do.

Data Plane (what PSE keeps local)
The actual user traffic — web pages, file downloads, inspected payloads, DLP checks. Processed entirely on-premises by the PISE or ZPA PSE. Never leaves the customer's network boundary.
Control Plane (what stays in the cloud)
Policy configuration, admin portal, threat intelligence feeds, software updates, and logs. These still go to Zscaler's cloud — but contain no user traffic content. This is the compliance-safe split.
The PSE formula: Zscaler features + customer location. Control plane in the cloud. Data plane on-premises. Traffic content never leaves the customer's network.
KEY CONCEPTS FROM THIS SESSION

What to carry forward

The Firewall Module's Four Engines

DPI Engine
Identifies the application from the first 1–5 packets. Catches apps on non-standard ports. Caches results for future sessions so no manual policy is needed.
DNS Engine
Header-based restrictions and domain-level access control for plain DNS (UDP/TCP 53). Does not apply to DoH — that rides on HTTPS and hits the Proxy Module.
IPS Engine
Per-packet threat detection via signatures. Alerts and blocks intrusion attempts across non-web flows.
Policy Engine
Evaluates firewall rules top-down, first-match. Controls what the other three engines enforce and when.

DNS vs DoH — The Gotcha

!
WATCH OUT — DNS vs DNS over HTTPS
Plain DNS (UDP 53) → Firewall Module → DNS Engine. No SSL inspection needed.

DNS over HTTPS (DoH) → rides on HTTPS port 443 → Proxy Module. SSL inspection can apply. Zscaler treats DoH as a visibility risk — it hides DNS lookups inside encrypted traffic, bypassing the DNS Engine unless SSL inspection is active.

SSL Inspection — The Dependency Chain

Multiple Day 4 features depend on SSL inspection being enabled. If you can't see inside the TLS tunnel, you can't enforce these capabilities:

  • Tenant Restrictions — needs to inject HTTP headers inside HTTPS requests
  • URL Filtering on HTTPS — needs to read the full URL, not just the domain
  • DLP inline — needs to inspect payload content
  • Cloud App Control activity controls — needs to see inside the app's requests
  • ATP / malware inspection on downloads — needs to see the file content
The rule: Any feature that needs to read or modify content inside HTTPS requires SSL inspection. DNS Control is the exception — it works on plaintext DNS regardless.
