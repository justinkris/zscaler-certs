What you can rely on

ATP + Cloud Sandbox Layered Logic ✓ SOLID

You correctly identified that ATP and Cloud Sandbox are distinct, sequential layers — not the same thing. ATP handles fast, cheap checks first; Sandbox is selective and triggered by risk score.

ATP — Advanced Threat Protection
Operates in 5 layers: URL Filtering → Content-based blocking → Reputation → IPS Signatures → ML/AI Analysis. Each layer is progressively more expensive. ATP verdicts happen inline, in real time.
Cloud Sandbox
Detonates files in an isolated environment to observe behaviour. Triggered selectively — only when ATP's upstream checks produce an uncertain or elevated risk score. Not applied to every file.

Step-Up Authentication ✓ SOLID

You correctly identified that step-up is triggered by a change in application context — specifically an application configured to require a higher authentication level than the user's current session provides.

Risk360 Pillars ✓ SOLID

Inline CASB vs API-based CASB ✓ SOLID

Deep Trace vs Cloud Path ✓ SOLID

Deep Trace = on-demand, real-time, per-user diagnostic. Cloud Path = continuous, aggregated path monitoring across users over time.

Authentication Levels 1 / 2 / 3 ⚠ GAP

The three levels are a ZPA construct used in Authentication Policy to define how strongly a user must prove their identity before accessing an application.

LevelRequirementWhen to use
Level 1Password onlyLow-sensitivity applications
Level 2Password + MFAStandard corporate applications
Level 3Password + MFA + Managed deviceHighly sensitive / regulated applications
Mental model: Airport security lanes. Level 1 = standard queue. Level 2 = security screening. Level 3 = screening + passport + biometric. The application decides which lane you must use.

Device Posture Profiles in ZPA ⚠ GAP

You understood the concept (device must meet security requirements) but couldn't recall the feature name. In ZPA, a Device Posture Profile defines the criteria a device must satisfy — OS version, certificate, ZCC status — and is referenced as a condition in the Access Policy.

DEVICE POSTURE PROFILE
A named set of device compliance criteria in ZPA. Referenced in Access Policy conditions. Think of it as the bouncer's checklist — the Access Policy is the door rule, the Device Posture Profile is what gets checked.

PISE — ZIA Private Infrastructure Service Edge ⚠ GAP

When ZIA enforcement needs to run within an organisation's own infrastructure (e.g. for latency reasons), the component is called a PISE — Private Infrastructure Service Edge. ZPA calls its equivalent a PSE. Don't confuse the two.

ZIAPISE — Private Infrastructure Service EdgeInternet-bound traffic inspection

OneAPI vs Workflow Automation — Direction of Integration ⚠ GAP

These two capabilities are easy to conflate but solve different problems. The key is direction.

CapabilityDirectionUse case

Workflow AutomationZscaler → External systemZDX alerts trigger automatic tickets in ServiceNow, etc.
Mental model: OneAPI is the front door — external systems knock and come in. Workflow Automation is the alarm system — Zscaler rings the external bell when something happens inside.
KEY CONCEPTS & ANALOGIES

Mental models from today

⟳
ATP → SANDBOX FUNNEL
ATP is a series of progressively expensive filters. Each layer narrows the pool of suspicious traffic. Only files that survive all upstream checks with an uncertain verdict reach the sandbox — because detonating everything would be prohibitively slow and resource-intensive. Think of it as a hiring funnel: most candidates are screened out at the CV stage; only the shortlisted few get the full interview.
⟳
AUTHENTICATION LEVELS — AIRPORT SECURITY LANES
Level 1 is the standard queue (just show your boarding pass). Level 2 adds full security screening (MFA). Level 3 requires biometric passport check too (MFA + managed device). The application at the gate decides which lane you must use — not you.
⟳
ONEAPI VS WORKFLOW AUTOMATION — FRONT DOOR VS ALARM SYSTEM
OneAPI is the front door: external tools knock, authenticate, and come inside to query or configure. Workflow Automation is the alarm system: when something happens inside Zscaler, it rings an external bell (ServiceNow, etc.) automatically — no one had to knock.
⟳
INLINE CASB VS API CASB — CUSTOMS vs POST OFFICE INSPECTION
Inline CASB is customs at the border — it inspects everything passing through in real time. API CASB is the post office inspecting packages already in the warehouse — it reaches into the SaaS app directly, out-of-band, to check what's already sitting there.
EXAM TIPS

What to watch for on exam day

→
EXAM TIP — TERMINOLOGY PRECISION
The exam loves testing node names: ZEN (ZIA) vs PSE (ZPA) vs PISE (ZIA private). These look similar and are easy to mix up. Memorise the product-to-name mapping cold.
→
EXAM TIP — DIRECTION OF INTEGRATION
Scenario questions about automation will describe a flow and ask which capability enables it. Always ask: who initiates? If Zscaler pushes outward (alert → ticket) → Workflow Automation. If an external system pulls or pushes inward → OneAPI.
!
WATCH OUT — ATP ≠ SANDBOX
A common distractor: questions that imply ATP "sends files to the sandbox." ATP is its own engine with its own detection layers. The sandbox is a separate, downstream service. Don't conflate them.
!
WATCH OUT — STEP-UP TRIGGER
Step-up is not triggered by a "failed login" or "suspicious behaviour." It is triggered by the application requiring a higher authentication level than the user's current session provides. The trigger is policy-driven, not anomaly-driven.
  • Authentication Level 3 requires both MFA AND a managed/compliant device — not just MFA alone
  • Device Posture Profile is the feature name in ZPA — not "device compliance policy" or "endpoint check"
  • PISE is ZIA-specific — don't use PSE when asked about ZIA's private enforcement node
  • Risk360 has exactly four pillars — Attack Surface, Compromise Likelihood, Lateral Movement, Data Loss Exposure
  • Deep Trace is on-demand and per-user; Cloud Path is continuous and aggregated
