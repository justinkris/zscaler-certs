THE ONE I MISSED — Q6

ZIA Inspection Pipeline Order

A file is downloaded over HTTPS. ZIA processes it through a sequence of inspection stages. The exam expects you to know the canonical stage order — the order Zscaler documents in the official EDU 202 study material.

STAGE 1
SSL Decryption
Open the bag — you can't inspect what you can't see.
STAGE 2
IPS
Metal detector — fast, network-level pattern checks.
STAGE 3
ATP / AV
X-ray — file/content signature scan against known malware.
STAGE 4
Sandbox
Secondary inspection — detonate unknown files in a VM.
⟳
AIRPORT SECURITY ANALOGY
The pipeline is a security funnel. Cheap, fast checks first; slow, expensive ones last. SSL decryption opens the bag. IPS is the metal detector you walk through. ATP/AV is the X-ray of the bag's contents. Sandbox is the room where they actually open and examine the suspicious item. You wouldn't send everyone to the back room — only what survives the earlier checks goes there.
!
WATCH OUT
The trap on Q6 was placing ATP before IPS. Plausible-sounding, because "advanced" feels heavier than "intrusion prevention". But the layering is by cost and depth, not by name. IPS sees the network stream (cheaper). ATP/AV needs the reconstructed file (more expensive). Sandbox actually runs the file (most expensive — always last).

Each stage by what it sees

Stage What it sees What it catches
IPS The network stream / packets Protocol exploits, network attack signatures
ATP / AV The reconstructed file / content payload Known malware, malicious payloads (signature)
Sandbox The file running in a VM Zero-day, unknown, behaviour-based threats
★
MENTAL HOOK
"Decrypt → IPS → ATP → Sandbox" — say it out loud three times before the exam. If a question shows ATP before IPS or Sandbox not last, it's wrong.
CYBERTHREAT RECAP

Cyberthreat Protection — Concepts I Confirmed

Quick reference for revision — not new material, just the most testable framings.

ATP vs IPS — the right framing
ATP protects against threats like botnets, phishing, command-and-control, and malicious active content (browser exploits, suspicious scripts). IPS detects and blocks network-based exploits and protocol-level attacks. They are complementary, not competing.
Cloud Sandbox
Detonates unknown files in an isolated VM to determine behaviour before the verdict is rendered. Configurable by file type and user group; can quarantine a file until the verdict comes back. Always the most expensive stage — runs only on files that weren't already blocked or cleared by signature.
Browser Isolation
Renders a risky session inside the Zscaler cloud and streams pixels to the user — no active code touches the endpoint. Can be policy-driven (auto-isolate uncategorised or risky sites) and can restrict download / copy-paste / print for BYOD scenarios.
Deception vs ITDR — easy to confuse
Deception plants decoy assets, credentials, and files to lure and detect attackers who have already bypassed perimeter controls — it's a tripwire for active intruders. ITDR focuses on identity hygiene: scans Active Directory and identity systems for misconfigurations, weak credentials, and risky trust relationships, and detects identity-based attacks.
Breach Predictor
Forward-looking, AI-driven capability that simulates and forecasts potential breach paths based on current posture and telemetry. Not historical reporting, not pen testing — predictive modelling.
Private Access AppProtection
A WAF-style inline inspection layer for private applications accessed via ZPA. Protects against OWASP Top 10 and similar application-layer threats. Sits in the ZPA path, not the ZIA path.
C2 / botnet defence — layered
ATP (botnet/C2 categories) + DNS Control + Firewall + IPS together. No single control owns this — exam answers that mention only one are usually wrong.
DATA PROTECTION RECAP

Data Protection — Concepts I Confirmed

Inline DLP vs SaaS API CASB

Aspect Inline DLP / Inline CASB SaaS API CASB (Out-of-Band)
When it acts Real time, in the proxy path Asynchronously, via SaaS API
What it inspects Data in motion (uploads, posts, transfers) Data at rest already in the SaaS
Requires SSL inspection? Yes — to see HTTPS payloads No — uses authenticated API
Use case Block sensitive uploads as they happen Scan historical content, retroactive policy

DLP detection techniques

Technique What it does When to use it
Dictionaries + Engines Patterns + logic (proximity, count, confidence) General-purpose detection (PCI, PII)
EDM — Exact Data Match Matches exact values from a known list (e.g. specific SSNs) Protecting a defined set of records
IDM — Indexed Document Match Fingerprints whole documents and detects derivatives Protecting known sensitive documents
OCR Reads text inside images / screenshots / scans Photographed IDs, scanned forms, screenshots
→
BYOD SCENARIO PATTERN
When the question says "personal device" or "BYOD" and "allow access but prevent download/copy/print" — the answer is almost always Browser Isolation with restricted controls. Don't get pulled into "block with URL filtering" — the user still needs access.
→
SSL INSPECTION IS A PREREQUISITE
If a DLP rule "works on HTTP but misses HTTPS", the cause is almost certainly that SSL inspection isn't enabled or isn't applied to that traffic. DLP can only inspect payloads it can see in cleartext. This is a classic exam scenario.
Endpoint DLP
Extends DLP enforcement to local actions on the device — USB transfers, printing, clipboard — even when the user is offline. Complements inline DLP rather than replacing it.
Tenant Restrictions
Allows access to a SaaS app (e.g. Microsoft 365) only for the organisation's own tenant and blocks sign-ins to personal or third-party tenants of the same app. Stops users from logging into "other" M365 instances from corporate devices.
Inline DLP decision flow
SSL decryption → content extraction → DLP engine evaluation (dictionary / EDM / IDM / OCR as configured) → policy action (allow / block / quarantine / notify) → incident logged with metadata.
Cloud App Control / CASB inline
The control plane for shadow IT — visibility into what SaaS apps are being used, plus policy to allow / restrict / coach / block based on app risk score. This is the right answer when the scenario mentions "unsanctioned SaaS" or "shadow IT".
EXAM TIPS

What this mock revealed about the real exam

→
ORDERING QUESTIONS ARE TRAPS
The single question I missed was an ordering question. Whenever the exam asks about the order of operations in a pipeline, slow down. The pattern Zscaler uses is "cheap and broad first, expensive and narrow last". Decrypt → fast network check → file signature check → expensive behavioural check.
→
"MOST COMPLETE" ANSWERS
When a question asks "how does Zscaler protect against X" — the right answer is almost always the one that lists multiple layers (e.g. ATP + URL filtering + SSL inspection + Browser Isolation for phishing). Single-control answers are distractors.
→
SCENARIO PATTERN MATCHING
Most Data Protection questions are scenario-driven. Train yourself to spot the keyword that pins the answer:
· "Personal device / BYOD" → Browser Isolation with restricted controls
· "Already in SaaS / data at rest" → SaaS API CASB
· "Real-time upload block" → Inline DLP
· "Other tenants of M365" → Tenant Restrictions
· "Image / screenshot / photo of doc" → OCR
· "Specific known records" → EDM
· "Whole sensitive document" → IDM
→
DECEPTION vs ITDR
One sentence each:
· Deception = decoys to catch active attackers in the network.
· ITDR = identity hygiene + detecting identity-based attacks.
If the question mentions "decoy / honeypot / lure" → Deception. If it mentions "Active Directory / identity / credentials" → ITDR.
