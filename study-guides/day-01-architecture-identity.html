OVERVIEW

Why the old model broke — and what replaced it

Traditional network security assumed that location equals trust. Inside the corporate network meant trusted. Security lived at the perimeter — firewalls, VPNs, a hard boundary between "inside" and "outside." This worked when users sat at desks in office buildings and apps lived on on-premises servers.

That model collapsed as users went remote and applications moved to SaaS platforms like Microsoft 365 and Salesforce. The perimeter became impossible to define. Zscaler's architecture responds directly: instead of securing a network boundary, it secures the connection between a user and a specific application — regardless of where either one lives.

Central insight: When you can no longer control where users and apps are, you stop trusting location and start trusting verified identity. Security moves from the network edge into the cloud, inline between every user and every destination.
CORE CONCEPTS

How Zscaler's architecture works

Zero Trust Network Access (ZTNA)

Zero Trust is built on one principle: never trust, always verify. No user or device is trusted by default — not even on the corporate network. Access is granted only after verifying identity, device posture, and context, and only to the specific application requested — never the network itself.

Zero Trust Network Access (ZTNA)
A security model where users are granted access to specific applications only — never the underlying network. Every request is verified regardless of source location. Replaces the VPN model of broad network access.
Least Privilege
Users receive only the access they need for their role — nothing more. Minimises the "blast radius" of a compromised credential: an attacker gains access to one specific app, not the entire network.

VPN vs ZTNA — the lateral movement problem

A VPN grants a remote user access to the corporate network, from which they can reach many applications. ZTNA grants access only to a specific application. If an attacker compromises a VPN credential, they can move laterally across the network. With ZTNA, a compromised credential only unlocks one door — and nothing else.

FeatureTraditional VPNZTNA (Zscaler)
What access is grantedBroad network accessSpecific application only
Trust modelLocation-based (inside = trusted)Identity-based, continuously verified
Lateral movement riskHigh — attacker can roam networkEliminated — no network access granted
Scales with remote/cloudNo — degrades as users leave the officeYes — cloud-delivered, location agnostic

Multi-tenant cloud architecture

Zscaler operates as a multi-tenant cloud platform distributed across enforcement nodes globally. Many organisations share the same underlying infrastructure, but each organisation's policies, data, and configurations are completely isolated. Customers get global scale, elastic capacity, and continuous updates — without building or maintaining their own hardware.

Multi-tenancy
Multiple organisations share the same cloud infrastructure but remain logically isolated. Shared plumbing, private living spaces. Enables massive scale without per-customer hardware investment.
Zscaler Enforcement Nodes (ZENs)
Globally distributed cloud nodes that sit inline between users and their destinations. All traffic routes through these nodes for inspection and policy enforcement. Global distribution ensures performance at scale.
MENTAL NOTES

Things to remember

★
MENTAL NOTE
The core shift is from location as the trust signal to identity as the trust signal. Everything about Zero Trust flows from that one swap. When a question describes a traditional perimeter model failing, the answer always involves moving trust to identity.
★
MENTAL NOTE
"Continuous verification" is not just login. It means every request, every session, re-evaluated against identity, device posture, and context. This is what "never trust, always verify" actually means operationally — not a one-time gate, but an ongoing check.
!
WATCH OUT
Multi-tenancy means shared infrastructure, not shared data or policies. Each customer's configuration is completely isolated. Exam questions may try to imply multi-tenancy creates a privacy risk between customers — it does not. Don't fall for it.

But when workers leave the castle and treasures move to the cloud, the moat protects nothing. Zscaler removes the castle entirely — security follows the user and the app, wherever they are.

Zero Trust continuous verification is like a nightclub with a bouncer at every internal door — not just the entrance. Getting past the front door doesn't grant access to the VIP room, the bar, or backstage. Each door requires a fresh check. Authenticating once at login doesn't open everything.

IDENTITY SERVICES

What is Identity Services?

Before Zscaler can inspect traffic, apply policies, or make any access decision, it needs to know one thing: who is this user? Identity Services is the set of mechanisms that establish and verify user identity at every point in the Zscaler platform.

Zscaler does not store identity itself — it delegates that responsibility to an external Identity Provider (IdP) such as Entra ID or Okta. This makes Zscaler fundamentally stateless with respect to identity: it trusts assertions from the IdP rather than managing sessions directly.

Core insight: Zscaler is the bouncer. The IdP is the VIP agency with the list. The bouncer never decides who's on the list — he just checks the signed credential the agency provides.
SAML FLOW

SP-Initiated SAML Authentication

The most common flow is SP-initiated — the user tries to reach a resource, Zscaler (the Service Provider) redirects them to the IdP for authentication, and the IdP sends back a signed assertion.

Step 1 — User requests access
Browser hits a resource protected by Zscaler — e.g. a ZPA app or internet traffic through ZIA.
Step 2 — Redirect to IdP
Zscaler sees no valid assertion. Sends a SAML AuthnRequest to the IdP (e.g. Entra ID). This request can include an authnContextClassRef specifying the required authentication level.
Step 3 — User authenticates
The IdP checks its SSO session cookie. If the user is already authenticated at the required level, no password prompt. If not, the IdP challenges them.
Step 4 — IdP issues SAML Assertion
Assertion is signed with the IdP's private key. Contains user identity, group memberships, attributes, and the authentication method used.
Step 5 — Zscaler verifies and enforces
Zscaler verifies the assertion signature using the IdP's public certificate (pre-configured). Now it knows who the user is and can evaluate policies.

Trust Exchange — What Gets Configured Up Front

IdP → Zscaler
The IdP's signing certificate is uploaded into Zscaler so it can verify assertion signatures at runtime.
Zscaler → IdP
Zscaler's Entity ID and ACS (Assertion Consumer Service) URL are registered in the IdP so it only responds to trusted requests.
Zscaler never holds the SSO session. The cookie lives with the IdP. Zscaler is stateless — it receives a fresh assertion each time, but the user never notices because the browser handles the roundtrip invisibly.
STEP-UP AUTHENTICATION

Demanding Stronger Proof Mid-Session

Step-Up Authentication says: for this resource, the current authentication level is not enough — prove yourself again, at a higher level.

LevelMethodStrength
Level 1Password onlyWeakest — phishable, reusable
Level 2Password + SMS / Authenticator appBetter — but SMS can be SIM-swapped
Level 3Hardware key (YubiKey) / Certificate-basedStrongest — cryptographic, physical possession required

When Zscaler redirects to the IdP for step-up, it includes an authnContextClassRef inside the SAML AuthnRequest. This tells the IdP: "I need authentication at this specific level — do not just accept the existing SSO cookie if it doesn't meet this bar."

Policy Evaluation Order

Zscaler always evaluates Authentication Policy before Access Policy. Identity must be resolved before access can be granted — Zscaler can't match group memberships or user rules against an unknown identity.

!
WHY ORDER MATTERS
If Access Policy ran first, Zscaler wouldn't know who the user is yet. Identity must always be resolved before access can be evaluated. This is a reliable exam signal — any answer that skips identity resolution is wrong.
LDAP AUTHENTICATION

Legacy Identity — When There's No Modern IdP

LDAP validates credentials directly against on-premises Active Directory using an LDAP BIND request — a binary yes/no. It is ZIA only; ZPA requires rich identity assertions (groups, attributes) that BIND cannot provide.

DimensionSAMLLDAP
Password visibilityZscaler never sees itZscaler receives it directly
Group membership freshnessFresh in every assertionOnly as fresh as last sync
On-prem infrastructureNoneAuthentication Bridge required
ZIA supportYesYes
ZPA supportYesNo
!
SYNC STALENESS RISK
If a user is terminated Monday but the next LDAP sync isn't until Tuesday, Zscaler's cached directory still shows them as active. They may still authenticate via BIND during that window — up to 24 hours of exposure. SAML has no such gap: user status is reflected in every fresh assertion.
IDENTITY PROXY

Zscaler as the IdP for SaaS Applications

Identity Proxy inverts the SAML relationship. Zscaler becomes the IdP for certain SaaS applications — sitting in the middle of every authentication attempt to that app, regardless of device or network.

The problem it solves: A user on an unmanaged device goes directly to salesforce.com, bypassing Zscaler entirely. Identity Proxy closes this gap — if Salesforce is configured to use Zscaler as its IdP, authentication must flow through Zscaler. There is no bypass route.

Supported apps: Box, GitHub, Google Apps, Microsoft O365, Salesforce, ServiceNow, ShareFile, Slack.

Policy actions for unmanaged devices: Block (deny entirely) or Browser Isolate (allow via isolated browser with no local download/copy/paste).

OIDC AUTHENTICATION

OpenID Connect — Modern SSO for APIs and Apps

OIDC uses JSON Web Tokens (JWTs) instead of XML assertions — making it native to modern APIs, mobile apps, and CLI tools that weren't designed around browser redirects.

DimensionSAMLOIDC
Token formatXML assertionJSON Web Token (JWT)
Best forBrowser-based enterprise web appsAPIs, mobile, CLI, developer tools
Trust credentialsEntity ID + ACS URL + certificateClient ID + Client Secret
ZIA / ZPA supportBothBoth

OIDC uses three separate IdP endpoints: Authorization (user challenge), Token (server-to-server code exchange), and UserInfo (extra attributes if scopes beyond "openid" are configured). JIT provisioning creates accounts automatically on first login — no pre-provisioning required.

→
WHEN TO USE EACH PROTOCOL
LDAP = ZIA only, legacy on-prem AD. SAML = browser SSO, XML, mature IdP. OIDC = modern apps, APIs, JWTs. If an exam question involves a CLI tool, mobile app, or API integration, the answer is OIDC. If it involves ZPA, eliminate LDAP immediately.
IDENTITY ANALOGIES

Ways to think about identity

⟳
THE GOVERNMENT PASSPORT
A SAML assertion is like a passport. The IdP is the government that issued it — signed with a seal no one can forge. Zscaler is the border agent who checks it. The agent doesn't call the issuing government on every crossing — they just verify the signature. Step-up authentication is like a customs officer asking for a secondary check before you enter a restricted zone.
⟳
THE HOSPITAL VISITOR LIST
LDAP sync is like a hospital's visitor list printed each morning. Security can check who's on the list — but if someone was removed at 2pm and the list isn't reprinted until tomorrow, they can still walk in. SAML is the live system at the desk: it checks in real-time with the registry on every entry.
⟳
THE HOTEL CONCIERGE
Identity Proxy is like a hotel concierge who stands between guests and every service in the building. If you want to access the spa, you go through the concierge — they verify who you are via the real IdP and check your membership level. Someone trying to sneak in through a side entrance still has to deal with the concierge, because the spa was configured to only accept bookings through him.
IDENTITY EXAM TIPS

Identity Services — What the exam tests

→
LDAP SCOPE
LDAP is ZIA only — never ZPA. If a ZPA scenario offers LDAP as an option, eliminate it immediately. ZPA requires rich identity context that LDAP BIND cannot provide.
→
IDENTITY PROXY BYPASS PREVENTION
Identity Proxy works because the SaaS app is configured to trust Zscaler as its IdP. Even if a user bypasses ZCC entirely, they cannot bypass Salesforce's IdP configuration. The control is enforced at the SaaS app level, not the network level.
  • Zscaler is stateless with respect to identity — it trusts assertions, never stores sessions.
  • SAML trust setup requires: IdP certificate in Zscaler, and Zscaler Entity ID + ACS URL registered in the IdP.
  • SSO works because the browser presents the IdP's session cookie silently — Zscaler is not involved in that session.
  • LDAP sync freshness is a security risk — termination creates a window of continued access up to 24 hours.
  • Identity Proxy supported apps: Box, GitHub, Google Apps, Microsoft O365, Salesforce, ServiceNow, ShareFile, Slack.
  • Authentication Policy runs before Access Policy — always resolve identity first.
