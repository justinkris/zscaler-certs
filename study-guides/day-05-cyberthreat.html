FOUNDATION

The prerequisite for everything

Nearly every Cyberthreat Protection capability in ZIA depends on one thing being enabled first. Without it, Zscaler can only see the outer envelope of encrypted traffic — source, destination, and TLS handshake details — but nothing inside.

SSL Inspection is the prerequisite for ATP, IPS, Cloud Sandbox, and DLP to function. Without it, all inspection engines are blind to content inside HTTPS traffic. It is not optional — it is the foundation.
ADVANCED THREAT PROTECTION

ATP — The inspection pipeline

ATP is Zscaler's layered engine for detecting and blocking malicious files, URLs, and destinations. It operates as a tiered pipeline — cheapest and fastest checks first, most expensive last — to balance security with scale.

1
Threat Intelligence — Known Bad Destination checked against Zscaler's global database of known malicious URLs, domains, and IPs. Instant block — no content inspection needed.
2
Heuristics & ML — Unknown but Suspicious Domains not in the database are analysed for suspicious signals: high entropy names (DGA), newly registered age, dormant-then-suddenly-active behaviour. AI/ML scoring flags risk.
3
Content Inspection — Response Analysis SSL-decrypted response content inspected. Hash/signature matching first — has Zscaler seen this file before? Then heuristic analysis for file type mismatches and structural anomalies.
4
Cloud Sandbox — Detonation Truly unknown files are executed in an isolated environment. Behaviour is observed — registry modifications, C2 callouts, OS changes. File bytes withheld from user until verdict.

Cloud Sandbox — Key Details

Patient Zero Protection
The connection is held open but file bytes are withheld while sandbox analysis runs. Nothing touches the endpoint until a clean verdict is returned. If malicious — connection dropped.
Sandbox Evasion Resistance
Sophisticated malware detects sandbox environments and goes dormant. Zscaler mimics realistic endpoint environments — hardware fingerprints, user activity patterns, installed software — to fool evasion techniques.
ModeBehaviourUse Case
Block Mode File withheld until sandbox verdict. Maximum protection. High-security environments — patient zero protection priority
Monitor Mode File released immediately. Analysis runs in parallel. Alert if malicious. Productivity-sensitive environments — latency cannot be tolerated
INTRUSION PREVENTION SYSTEM

IPS — Behavioural traffic inspection

IPS uses deep packet inspection against a signature database to detect attack patterns in the traffic stream — specifically in inbound response traffic coming back to users from the internet.

ATP asks: "Is this file or destination malicious?"
IPS asks: "Is this conversation behaving like an attack?"

IPS catches what ATP misses. A URL and file can both appear clean, but as more packets arrive, the pattern of the conversation can betray an attack — timing anomalies, payload structures matching exploit kits, unexpected callbacks.

What IPS detects

Cross-Site Scripting (XSS)
Malicious scripts injected into legitimate page responses. Executes in the user's browser without any user interaction.
Exploit Kits
Malicious code served in a webpage that probes for browser/plugin vulnerabilities and automatically delivers a payload if one is found.
C2 Communication Patterns
Traffic matching known command-and-control communication signatures — timing, frequency, and payload structure of beacon callbacks.
⟳
ANALOGY
ATP is the border officer checking your luggage for contraband. IPS is the officer watching how you're behaving — suspicious patterns, criminal body language. Clean luggage doesn't mean innocent behaviour. Both are needed.
BROWSER ISOLATION

Remote Browser Isolation (RBI)

The problem with a normal browser is that it executes everything it receives — JavaScript, HTML, CSS — directly on the endpoint. Browser Isolation air-gaps the user's device from web content entirely by moving execution to the cloud.

Core principle: The browser executes in Zscaler's cloud. The user's device receives only a safe visual stream. No executable content ever touches the endpoint.
ModeHow it worksTradeoff
Streaming Isolation Live interactive video feed of cloud browser — like a remote screen share. Full interactivity. Higher bandwidth. Full user experience.
Pixel / DOM Isolation Page rendered to a flat visual stream — no HTML or JS sent. Drawing instructions only. Lower bandwidth. Limited interactivity.
DeploymentWhere browser runsWhen to choose
Cloud Browser Isolation Zscaler's cloud infrastructure General internet browsing — easy deployment, no on-prem needed
Local Browser Isolation Organisation's own infrastructure Data sovereignty, strict compliance — traffic cannot leave org environment
DECEPTION

Deception technology — Post-breach detection

Deception is designed for attackers who have already gotten past perimeter controls and are moving laterally. Unlike ATP or IPS, it does not need to judge whether something is malicious — interaction with a decoy is proof by definition.

Zero false positives: No legitimate user or system ever touches a decoy. Any interaction is definitionally a confirmed threat.

How it works — The trap

Zscaler plants lures (fake credentials, saved passwords) on real endpoints. Those lures point to decoys (fake servers) in Zscaler's deception cloud. DNS records for fake hostnames route attacker traffic to the deception cloud. The moment an attacker interacts — the compromised endpoint, user, technique, and timing are all immediately known.

Asset taxonomy

TypeWhat it isWhere it lives
LuresFake credentials/shortcuts planted on real endpoints — the breadcrumbsReal user machines
DecoysFake servers and resources that lures point toZscaler deception cloud
AD DecoysFake Active Directory objects — users, service accounts, groupsCustomer's Active Directory
Honey TokensFake files with embedded beacons that phone home when openedFile shares, endpoints

AD-specific attacks Deception & ITDR detect

DCSync
Attacker mimics a domain controller using MS-DRSR protocol to replicate credential hashes including krbtgt. Signal: a non-DC machine making AD replication requests.
Kerberoasting
Attacker requests Kerberos service tickets for service accounts, then cracks them offline. Signal: unusually high volume of service ticket requests from a single user.
Golden Ticket
Attacker uses stolen krbtgt hash to forge Kerberos tickets entirely. Signal: tickets used from unexpected locations or with abnormal lifetimes.
!
WATCH OUT
Deception is a post-breach detection control — not prevention. It catches attackers already inside moving laterally. Do not confuse it with perimeter controls like ATP or IPS.
IDENTITY THREAT DETECTION & RESPONSE

ITDR — Identity is the new perimeter

In a Zero Trust architecture, identity is the primary access control mechanism. A stolen credential is a stolen perimeter. ITDR detects when identities — human or non-human — are being abused, even when authentication technically succeeds.

ITDR watches behaviour, not just authentication. A valid login from an impossible location at an unusual time accessing resources never touched before — is not a valid session.

Behavioural signals ITDR monitors

Location anomaly
Login from a country or region the user has never accessed from before.
Impossible travel
Two logins from geographically distant locations within a timeframe that makes physical travel impossible.
Time anomaly
Access outside the user's established working hours pattern.
Resource anomaly
Accessing sensitive systems or data the user has never touched before.

Automated response spectrum

SeverityResponse
LowAllow but monitor — increase logging and visibility
MediumStep-up authentication — challenge with MFA before continuing
HighRestrict access — limit to low-sensitivity resources only
CriticalSession termination or full account block

Severity thresholds and automated responses are defined by administrator policy. ITDR uses telemetry that flows naturally through Zscaler — no separate agents or SIEM required.

Non-human identities

ITDR also monitors service accounts, API keys, and application tokens. These are particularly dangerous when compromised — they often have broad permissions, never expire, and generate no human behavioural baseline.

PRIVATE ACCESS APPPROTECTION

AppProtection — WAF inside ZPA

ZPA controls who gets access to private applications. But once a legitimate authenticated user is connected, ZPA has no visibility into whether the requests they send are malicious. AppProtection fills this gap.

AppProtection is a WAF built natively into the ZPA traffic flow. It inspects HTTP requests to private applications inline — in the ZPA cloud — protecting against OWASP Top 10 attacks from authenticated users.

Architecture

Inspection happens in the ZPA cloud — not at the App Connector. Zscaler manages the inspection engine, signature updates, and rule sets centrally. The customer configures policy from the admin portal.

Traffic flow: User → ZPA Cloud (AppProtection inspects here) → App Connector → Private App

What it protects against

OWASP Core Rule Set
AppProtection rules align to the OWASP Core Rule Set — covering injection attacks (SQL, command), XSS, directory traversal, broken access control, and other OWASP Top 10 categories.
→
EXAM TIP
ZPA = access control (who gets in). AppProtection = request inspection (what they do once inside). These are complementary — not overlapping. Exam questions may ask which control applies to post-authentication threats against private apps. The answer is AppProtection.
