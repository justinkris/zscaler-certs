OVERVIEW

What Day 4 covers

Day 4 covers two exam domains that together represent 19% of the blueprint: Platform Services (9%) and Access Control Services (10%). These topics describe the chassis of the Zero Trust Exchange — the hardware and virtual infrastructure that extends Zscaler into your environment, the analytics that make it observable, and the policy framework that enforces every access decision.

Everything here is foundational to the rest of the ZDTE exam: understanding policy evaluation order, PSE architecture, and the Tenant Restrictions / Conditional Access model underpins almost every scenario question you'll see in ZIA, ZPA, and ZDX domains.

Day 4 in one line: Private Service Edges extend the Zero Trust Exchange into your DC; Analytics and SIEM streaming make it observable; the Policy Framework, Firewall/DNS, Tenant Restrictions, Segmentation and Conditional Access are the layered controls that enforce Zero Trust on every request.
SUB-TOPIC 1 · PRIVATE SERVICE EDGE

Zscaler Private Service Edge (PSE)

A Private Service Edge is a Zscaler-managed extension of the Zero Trust Exchange running on customer infrastructure — either on a Zscaler-shipped appliance or as a virtual machine. It delivers the same inspection and brokering capabilities as a Public Service Edge, but locally.

Why PSE exists — the three drivers

  • Data sovereignty / geopolitical: Certain regulations require that traffic be inspected and logged in-country, on infrastructure the customer controls.
  • Latency / hairpin avoidance: Keeps local traffic local — e.g., users in a Sydney DC accessing Sydney-hosted apps don't hairpin through a distant Public Service Edge.
  • Source IP preservation: Egressing traffic through the customer's public IP block so that partner IP allowlists, licensing-bound SaaS, and geo-restricted services continue to work.

Two types — ZIA PSE vs ZPA PSE

ZIA Private Service Edge
Inspects internet-bound traffic. Same proxy/firewall/DLP/sandbox capabilities as a ZIA Public Service Edge. Communicates back to the ZIA Central Authority, cloud routers, and Nanolog clusters.
ZPA Private Service Edge
Brokers private application connections. Manages sessions between Zscaler Client Connector and App Connectors. Enforces ZPA access policies, caches path selection, and provides Universal ZTNA for on-prem users.

A single Service Edge plays one role — ZIA or ZPA. Not both. If an organisation needs both, they deploy both.

PSE as an extension of the Zero Trust Exchange

A PSE is not a standalone box — it's a remote node of the Zscaler cloud. It communicates with three key cloud components:

  • Central Authority (CA) — control plane; sends policy and authenticates users.
  • Nanolog clusters — logging plane; stores compressed transaction logs.
  • Cloud routers — route logs to the right tenant's Nanolog cluster. (Not traffic routers — log-pipeline routers.)

These three map to the three-plane architecture — data (Service Edges), control (CA), and logging (routers + Nanolog) — scaling independently so no single plane becomes a bottleneck.

Trust & enrollment

A PSE authenticates to the cloud via a provisioning key (secure random token tied to a tenant and group) and an enrollment certificate. On boot-up, the PSE presents its key, receives a signed cert from the cloud, and thereafter every communication is mutually authenticated via certificates.

Physical hardware models

Model Service Edges LB NIC Max Download
PSE 3 (Integrated LB) 3 instances 1 integrated 1GE ~1 Gbps
PSE 5 (Integrated LB) 5 instances 1 integrated 10GE ~2 Gbps
PSE 5 (Dedicated LB) 6 instances per node Up to 4 dedicated 10GE >5 Gbps
The >5 Gbps Rule: Any deployment requiring more than 5 Gbps of total throughput (upload + download combined) must use the Dedicated LB architecture. At high throughput, the integrated LB fights the Service Edge instances for CPU, and TLS decryption (the most expensive operation) degrades first.

Redundancy: minimum two PSEs per cluster — Zscaler doesn't support standalone deployments. Dedicated LBs also deploy in pairs.

Both the PSE hardware and the Dedicated LB are Zscaler-shipped, Zscaler-managed appliances. The customer does not supply their own load balancer.

Virtual PSE

PSE software also runs as a VM on VMware ESX/ESXi, Microsoft Hyper-V, AWS, Azure, GCP, and Enterprise Linux (RPM). ZPA PSE additionally supports Docker. Virtual deployment:

  • Eliminates hardware lead time (no shipping).
  • Leverages existing hypervisor infrastructure.
  • Unlocks deployment locations physical can't reach — notably inside public cloud regions (e.g., inspect Azure workloads in the same Azure region).
  • Lower peak throughput than physical (hypervisor overhead).
  • No Dedicated LB option — that's physical-only.

Deployment topologies

Topology Firewall impact Zscaler's stance
Inside DMZ Session doubling — every user transaction traverses the firewall twice, risking port and session table exhaustion Acceptable if security posture requires DMZ isolation
Outside corporate firewall (GRE/IPSec tunnel from firewall to PSE) One tunnel session on the firewall regardless of user volume Recommended — collapses N transactions into 1 session
Behind the firewall Firewall still in critical path; PSE-originated sessions still tracked Option, but not ideal
⟳
ANALOGY
Session doubling is the hotel front desk tracking every guest twice when they go out the side entrance. The tunnelled topology is like putting the whole department on a single shuttle bus through one checkpoint — one session to track, not thousands.
SUB-TOPIC 2 · ANALYTICS & REPORTING

Analytics & Reporting

The logging pipeline

USER TRANSACTIONS ↓ [Service Edge / PSE — generates logs] ↓ compressed + tokenised at source ↓ [Cloud Routers — route by tenant] ↓ [Nanolog Cluster — tenant-specific storage] ↓ ┌──────┴──────┬──────────────┐ ↓ ↓ ↓ [UI Server] [Data Lake] [Streaming] live queries long-term │ analytics ┌─────┴─────┬─────────────┐ ↓ ↓ ↓ NSS LSS Cloud-to-Cloud (ZIA) (ZPA) (either)

Efficient logging

  • Tokenisation & compression — payload reduction happens at the point of log creation, not in the cloud. NSS achieves up to 50:1 compression.
  • Log routers ensure each tenant's logs land in the correct Nanolog cluster.
  • Nanolog clusters provide the searchable store behind the admin UI.
  • Data lake enables long-term retention and analytics workloads beyond real-time use cases.

SIEM streaming options

SIEM location Product Method Transport
On-prem SIEM ZIA NSS (Nanolog Streaming Service) Customer-deployed VM in the DC, outbound TLS to Nanolog
On-prem SIEM ZPA LSS (Log Streaming Service) Via existing App Connector infrastructure
Cloud SIEM ZIA or ZPA Cloud-to-Cloud Log Streaming Direct HTTPS outbound connector — no customer VM required
Follow the destination. When choosing a streaming method, don't start by asking what you're streaming — ask where the SIEM lives. On-prem SIEM = you need a bridge into the customer network. Cloud SIEM = wire the two clouds together directly.

NSS specifics: customer-managed VM (Zscaler provides the image, customer patches and monitors it); outbound-only TLS to Nanolog cluster IPs; 1-hour buffer during connectivity issues; decompresses and reformats logs for the SIEM (syslog, JSON, CEF, LEEF). NSS and LSS follow the same design pattern as App Connectors — outbound connections only, no inbound firewall rules required.

SUB-TOPIC 3 · POLICY FRAMEWORK

Policy Framework — structure & hierarchy

Every policy in Zscaler — URL filtering, firewall rules, DLP, sandbox, tenant restrictions, ZPA access policies — follows the same underlying structure. Understanding the framework saves you memorising each policy type individually.

Universal rule anatomy

IF [criteria about the user / device / source] AND [criteria about the destination / object] THEN [action]

Every rule decomposes into three building blocks:

  • Subject criteria (who/where/how): user, group, department, location, device trust level, network, time window.
  • Object criteria (what they're accessing): URL category, cloud app, application segment, file type, threat category, content type.
  • Action: Allow, Block, Caution, Isolate, Quarantine, step-up authentication, etc.

First-match-wins evaluation

Rules are evaluated top-to-bottom. The first rule whose criteria all match fires — that rule's action is applied and no further rules are considered.

Design rule: Specific rules go on top. General rules go on the bottom. Exception first, general last. If a rule says "Allow CEO → anything" positioned below a "Block Social Networking" rule, the CEO still gets blocked from Facebook — the block rule matches first.

The pipeline model — layered evaluation

[Request arrives] ↓ Authentication who are you? ↓ URL Filtering is this site allowed? ↓ Cloud App Control is this SaaS app allowed? ↓ SSL Inspection decrypt (required for deeper layers) ↓ File Type Control is this file type allowed? ↓ IPS / ATP quick signature matching ↓ Malware full content scan ↓ Sandbox detonation of unknown files ↓ DLP is sensitive data leaving? ↓ Bandwidth Control QoS / shaping ↓ [Response returned]

A deny at any stage terminates the pipeline. A permissive "Allow CEO" URL rule does not bypass malware scanning — security inspection runs on allowed traffic, regardless of access policy outcome.

⟳
ANALOGY
Airport security: Authentication = check-in, URL Filter = ticket check, Cloud App Control = gate agent, Malware = bag X-ray, Sandbox = bomb squad room, DLP = customs on exit. Cheap checks early, expensive checks late. Each stage can stop the journey.
SUB-TOPIC 4 · FIREWALL & DNS CONTROL

Cloud Firewall & DNS Control

Cloud Firewall

ZIA's Cloud Firewall handles non-web traffic — SSH, RDP, DNS, custom TCP/UDP apps, anything that isn't HTTP/HTTPS. It runs in parallel with the proxy stack: at the Service Edge, a protocol classifier routes traffic to either the proxy pipeline (web) or the firewall pipeline (non-web).

TierCapabilities
Standard Cloud Firewall L3/L4 only — source/destination IP, port, protocol. Basic allow/block. No user or application awareness. Limited logging.
Advanced Cloud Firewall L7 application awareness, user/group/location-aware rules, IPS integration, full logging, NAT controls. Requires separate licensing.

Same framework as everything else: first-match-wins, specific-above-general. Cloud Firewall is always present on a ZIA deployment — licensing unlocks capabilities, not the module itself.

DNS Control

Policy-based control over DNS resolution. ZIA intercepts DNS queries and can allow, block, redirect, or inspect them. DNS is often the earliest signal in an attack (malware beaconing, C2 rendezvous, phishing kit domains), so blocking at DNS stops attacks before the TCP handshake.

Key features: DNS Gateways (defined endpoints for resolution), DNS filter rules, and DNS over HTTPS (DoH) inspection. Browsers and OSes increasingly default to DoH to hide queries in HTTPS; Zscaler detects and enforces policy on DoH traffic.

Order matters: DNS resolution happens before any TCP connection. So for non-web traffic, DNS Control evaluates first, then Cloud Firewall. For web traffic, DNS Control → proxy pipeline.
SUB-TOPIC 5 · TENANT RESTRICTIONS

Tenant Restrictions (v1 & v2)

Tenant Restrictions solves a gap URL filtering can't: the destination outlook.office.com is the same whether Bob signs into ACME's corporate M365 tenant or his personal Hotmail account. URL filter can't tell the difference.

Mechanism — HTTP header injection

Zscaler's proxy decrypts the HTTPS session (SSL inspection must be enabled), injects Tenant Restriction headers, and re-encrypts. Microsoft reads the headers at their endpoints and enforces the policy. Zscaler injects; Microsoft enforces.

v1 vs v2 — the core difference

Dimension TRv1 TRv2
What it controls Sign-in only Sign-in + resource access
Where policy lives Inside the injected header (allowlist) In Entra ID admin portal
Header format Plaintext tenant allowlist Signed JWT referencing tenant GUID
Anonymous access protection ❌ None ✅ Yes
Per-user / per-group rules ❌ No — tenant-wide only ✅ Yes
Per-application rules ❌ No ✅ Yes
To change allowed tenants Change Zscaler config Change Entra policy
One-line mental model: v1 is a bouncer at the club door. v2 is a bouncer at the door AND inside every room. v1 protects sign-in endpoints only; v2 protects sign-in AND resource endpoints (Graph, SharePoint, OneDrive, Teams, Exchange).

Scenario decision pattern

For any Tenant Restrictions exam question, ask two things in order:

  • Is this a sign-in or a resource access? Sign-in = both versions protect it. Resource access = only v2.
  • Where does the policy live and what does it say? v1: check the header allowlist. v2: check the Entra policy.
SUB-TOPIC 6 · SEGMENTATION & CONDITIONAL ACCESS

Segmentation & Conditional Access

Segmentation — "what can you reach?"

Segmentation in ZPA is application-level, not network-level — users don't get an IP address on the corporate LAN; they get brokered access to specific apps.

  • Application Segments — logical groupings of FQDNs / IPs / ports that make up one application.
  • Segment Groups — collections of Application Segments for easier policy targeting.
  • Server Groups — which App Connectors can reach which apps.
  • Access Policies — rules matching users/groups to Application Segments.

Everything is default-deny. Unauthorised users don't just get "access denied" — the app is invisible to them.

Conditional Access — "under what circumstances?"

Conditional Access layers context-aware conditions on top of segmentation. The policy engine evaluates:

  • Identity — user, group, role
  • Device posture — managed vs BYOD, Device Trust Level, compliance
  • Location — trusted network, country, IP
  • Time — business hours, unusual access times
  • Risk signals — ZDX score, threat score, behavioural anomalies
Segmentation + Conditional Access: Segmentation decides what rooms you can enter. Conditional Access decides when, where, and how you can enter them. Together they form the core of Zero Trust access control.
MENTAL NOTES

Things to remember

★
PSE IS AN EXTENSION
A Private Service Edge is a franchise location of the Zscaler chain. Customer provides the real estate (hardware/VM, rack, power, network); Zscaler provides the software, management, and policy. Cut its link to the cloud and it becomes stale — no policy updates, no auth refreshes, no logging.
★
THE THREE PLANES
Data (Service Edges) · Control (Central Authority) · Logging (Cloud Routers + Nanolog). Each scales independently. Great way to reason about failure modes: "What breaks if X plane loses connectivity?"
★
FIRST MATCH WINS
Every Zscaler policy list evaluates top-to-bottom and stops on the first match. Specific rules (exceptions) go above general rules. Getting the order wrong causes the CEO to be blocked from Facebook even though "Allow CEO → anything" exists further down.
★
PIPELINE, NOT SINGLE RULE
URL Filter allowing traffic does NOT bypass malware scanning. Each layer is independent, runs sequentially, and can terminate the request. "Allow" means different things at different layers.
★
ZSCALER INJECTS, MICROSOFT ENFORCES
For Tenant Restrictions, Zscaler's job is purely to inject the correct HTTP header on Microsoft-bound traffic. Microsoft's servers read the header and enforce the policy. Zscaler does not make the allow/block decision for tenant restrictions.
★
"FOLLOW THE DESTINATION"
For PSE type selection: public internet = ZIA PSE; private app = ZPA PSE. For SIEM streaming: on-prem SIEM = NSS (ZIA) or LSS (ZPA); cloud SIEM = Cloud-to-Cloud. Always let the destination drive the choice.
ANALOGIES

Ways to think about it

⟳
FRANCHISE RESTAURANT
PSE = a franchise restaurant on the customer's real estate. HQ (Zscaler cloud) sets the menu, recipes, policies, and branding. The provisioning key + enrollment cert = the franchise agreement. Daily sales reports to HQ = logs streamed to Nanolog.
⟳
KITCHEN AT DINNER RUSH
At low traffic, the greeter (LB) and chefs (Service Edges) can share a single kitchen. At high throughput, they compete for space and the chefs (who do the expensive TLS decryption) suffer first. Dedicated LB = moving the greeter out to a dedicated host stand. Same principle, better scaling.
⟳
SHUTTLE BUS THROUGH CHECKPOINT
DMZ PSE = every employee individually badges through security. Tunnelled (outside-firewall) PSE = the whole department rides one shuttle through one checkpoint. One session tracked vs. thousands. This is why the tunnelled topology is recommended.
⟳
AIRPORT SECURITY
Zscaler's policy pipeline is an airport: Authentication = check-in, URL Filter = ticket check, Cloud App Control = gate agent, File Type = bag type, Malware = bag X-ray, Sandbox = bomb squad room, DLP = customs on exit. Cheap checks early, expensive checks late, each stage can stop the journey.
⟳
BOUNCER AT A NIGHTCLUB
Tenant Restrictions = the bouncer checking IDs. TRv1 = bouncer has a printed list of allowed clubs on a clipboard — checks at the door only. TRv2 = bouncer's walkie-talkie calls head office (Entra) for dynamic rules, AND there's a bouncer inside every room (resource endpoint).
EXAM TIPS

What the exam tests

→
PSE MODEL SIZING
Expect throughput sizing questions. Memorise: PSE 3 ≈ 1 Gbps, PSE 5 Integrated ≈ 2 Gbps, PSE 5 Dedicated LB > 5 Gbps. The magic threshold is >5 Gbps total throughput requires Dedicated LB. Also: minimum 2 PSEs per cluster (no standalone).
→
STREAMING SERVICE IDENTIFICATION
Classic distractor: swapping NSS ↔ LSS. Remember — NSS streams ZIA logs via a customer-deployed VM; LSS streams ZPA logs via existing App Connectors. Cloud-to-Cloud is not limited to three named vendors — any SIEM reachable via HTTPS works.
→
RULE ORDERING
If the exam shows you a list of rules and asks what happens, trace evaluation top-to-bottom and stop on the first match. Specific → general. Exception → default. The CEO in the wrong position gets blocked with everyone else.
→
SSL INSPECTION PREREQUISITE
Tenant Restrictions, DLP, and Advanced Cloud Firewall inspection of encrypted flows all require SSL inspection to be enabled. No SSL inspection = these features silently fail to enforce. Expect at least one trick question about this.
→
TRv1 vs TRv2 SCENARIOS
Anonymous public links, Microsoft Graph calls, and cross-tenant resource access are all v2-only protections. v1 missed them entirely. The question "can this be expressed per-user or per-application?" is always answered by v2. If a question describes sign-in being blocked but anonymous file access succeeding — that's v1's blind spot.
→
VIRTUAL vs PHYSICAL PSE
Virtual PSE can deploy inside cloud regions (AWS, Azure, GCP) — physical cannot. Dedicated LB is physical-only. If a question describes a cloud region with no hardware allowed, virtual PSE is always the answer. If it asks about >5 Gbps on-prem, Dedicated LB physical is the answer.
