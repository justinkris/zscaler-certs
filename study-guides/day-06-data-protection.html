OVERVIEW

What is Data Protection Services?

Data Protection Services is the pillar of the Zscaler platform focused on preventing sensitive data from leaving your organisation — whether by web upload, SaaS sharing, endpoint action, or cloud storage exposure. It unifies DLP, CASB, DSPM, and Endpoint DLP under a single policy engine, enforcing the same detection logic across every channel sensitive data can travel through.

As an engineer, you're expected to understand how each channel is protected, when to use inline versus API-based enforcement, how DLP detection mechanisms combine, and how incidents are surfaced to users and admins.

Key point: Zscaler applies uniform DLP enforcement across all data channels — web, SaaS API, cloud storage, private apps, endpoints, USB, clipboard, and email — using the same DLP Engines, Dictionaries, and patterns. One policy engine, enforced everywhere.
DATA IN MOTION

SSL Inspection & DLP Inline

DLP inline only works if Zscaler can see the data. Since most traffic is TLS-encrypted, SSL inspection is a non-negotiable prerequisite. Zscaler acts as a trusted man-in-the-middle by presenting a dynamically generated certificate signed by Zscaler's own root CA, which must be installed in the trusted root store on every user device (typically via GPO, MDM, or endpoint management).

Because Zscaler is a full forward proxy, it holds the request in its entirety before forwarding it. This means inspection happens before data leaves the organisation — giving Zscaler the ability to block, modify, or redirect sensitive uploads in real time.

DLP Policy Actions

Allow
Traffic passes through with no interference. Used for low-risk scenarios.
Block
Traffic is stopped outright. Used for high-risk or clear policy violations.
Confirm Allow
Default outcome is to allow, but the user is prompted to justify the action first. A lighter-touch awareness nudge.
Confirm Block
Default outcome is to block; user must actively justify to proceed. Higher friction, used for riskier actions.
Exempted
Traffic matched a rule but was explicitly exempted (e.g., authorised user uploading to approved destination).

DLP Detection Mechanisms

DLP Dictionaries
Lists of sensitive words/phrases with a Match Count threshold to reduce false positives (e.g., trigger only if 3+ dictionary terms appear).
Pattern Matching (Regex)
Detects structured data — credit cards, SSNs, passport numbers, phone numbers. Supported by Confidence Score thresholds.
DLP Engines
The logic layer that combines dictionaries and patterns with Boolean expressions (AND/OR/NOT) and match counts.
OCR (Optical Character Recognition)
ML/AI-powered extraction of text from screenshots, images, and handwritten content. Closes the image-based evasion gap.
ML Categories
Trained classifiers that recognise categories of sensitive content (tax docs, medical records, source code) without explicit rules.
Microsoft AIP Integration
Reads classification labels applied by document authors, and can also write labels during data-at-rest scans.
SECURE SAAS DATA

CASB: Inline vs API-based

Dimension Inline CASB API-based CASB
Data state Data in motion Data at rest
Position In traffic path (proxy) Out-of-band via SaaS API
Timing Real-time, pre-delivery Scheduled or on-demand scans
Prerequisite SSL inspection OAuth / API integration with tenant
Blocks uploads? Yes — before data leaves No — data already in tenant
Remediation Block / Alert / Justify Quarantine, revoke sharing, change permissions, notify owner

What API-based CASB uniquely catches

  • Files shared via "anyone with the link" public URLs
  • Files shared externally to non-corporate domains
  • Overly permissive folder permissions
  • Sensitive content already at rest in SaaS tenants
  • Misconfigured tenant settings (weak password policies, MFA disabled)

Both detection modes share the same DLP engines — dictionaries, patterns, OCR, ML categories, AIP labels. You don't build two separate policies.

CLOUD & ENDPOINT

Secure Cloud Data and Endpoint Data

DSPM — Data Security Posture Management

Extends the API-based approach beyond SaaS to public cloud storage (AWS S3, Azure Blob, GCP Cloud Storage). Detects publicly accessible buckets, unencrypted sensitive data, weak access controls, and exposure paths. Provides root cause analysis and step-by-step remediation guidance.

Endpoint DLP via Zscaler Client Connector

Covers the channels that never touch the network. Enforced by ZCC on managed endpoints.

  • Web traffic — inline DLP
  • SaaS applications — CASB API
  • Public cloud storage — DSPM (AWS, Azure, GCP)
  • Private applications — ZPA DLP
  • USB / removable storage
  • Bluetooth file transfer
  • Clipboard copy-paste
  • Local file sharing
  • Corporate email (Exchange, Gmail)
→
EXAM TIP
The User Confirmation workflow is most commonly tested in the context of printing PII — the user is prompted to justify or cancel, creating an education moment rather than a hard block.

Outbound Email DLP

Extends DLP to email via SMTP mail flow rules with Microsoft Exchange. Zscaler scans outbound email, inserts policy headers, and Exchange enforces actions based on those headers.

UEBA & Adaptive Access

Detects anomalous behaviours: bulk uploads/downloads, impossible travel, failed login bursts. Can trigger MFA challenges or disable user access for high-risk activity.

BYOD ACCESS

Secure SaaS Access from BYOD

Unmanaged devices have no ZCC, no corporate root CA, and no way to enforce Endpoint DLP or inline inspection. The solution: never let the device actually hold the data.

⟳
ANALOGY
Browser isolation is like watching a movie through a window — you can see everything, but you can't reach through the glass to take it home. Only pixels are streamed to your device; the real app runs in Zscaler's cloud.

Browser Isolation for SaaS

The real browser runs in Zscaler's cloud; only pixels stream to the user's device. Downloads, clipboard, print, and right-click actions can be disabled at the cloud-browser layer. Screens can be watermarked with the user's name to deter screenshots.

Privileged Remote Access (PRA) for Private Apps

Agentless, browser-based access to private applications — SSH, RDP, HTTPS — brokered entirely in Zscaler's cloud. Ideal for third-party contractors and BYOD users who need internal app access without installing ZCC.

Scenario Solution
BYOD → SaaS apps Browser Isolation (via ZIA)
BYOD → Private apps PRA (with isolation controls)
INCIDENT MANAGEMENT

Incident Management Workflows

Zscaler doesn't try to be the single pane of glass — it pushes incidents to the tools your security team already uses.

User-facing notification channels

  • Browser-based pop-ups — branded with company messaging
  • Slack / Microsoft Teams — native integrations
  • Zscaler Client Connector pop-ups — desktop alerts

Admin-facing channels

  • Email notifications — violation metadata, match snippets, full payload attachments
  • NSS (Nanolog Streaming Service) — streams logs to SIEM (Splunk, QRadar, Sentinel)
  • ServiceNow / ITSM — automated ticket creation on violation
  • On-premises incident receiver — for data residency / regulated industries where payloads can't leave on-prem
★
MENTAL NOTE
Two audiences, two channels: Users get Slack/Teams/ZCC pop-ups for awareness + justification. Admins get Email/NSS/ServiceNow/on-prem receiver for response + audit trail.
EXAM TIPS

What the exam tests

→
EXAM TIP
Know the difference between Confirm Allow (default = allow, user nudged) and Confirm Block (default = block, user must justify). The default outcome is hidden in the name.
→
EXAM TIP
SSL inspection is the prerequisite for inline DLP. Many exam scenarios test "DLP isn't detecting" — the first thing to verify is always SSL inspection enabled for the destination domain.
→
EXAM TIP
Know the troubleshooting order: check Web Insights → verify SSL inspected = Yes → check DLP Engine match → verify Dictionary and Confidence Score → check Match Count logic.
!
WATCH OUT
When a user uploads multiple files in the same session, the DLP engine stops processing after the first policy match. So if file A triggers the rule, file B (uploaded in the same HTTP POST) won't be scanned.
!
WATCH OUT
Inline CASB and API-based CASB are complementary, not alternatives. A typical exam scenario tests whether you know which one handles a given risk — oversharing = API, active upload = inline.

MY NOTES
Confirm Allow vs Confirm Block — the default outcome is literally in the name. "Confirm Allow" = nudge, will allow by default. "Confirm Block" = justify to proceed.
Same DLP engines power inline, CASB API, Endpoint DLP, Email DLP. One policy, enforced everywhere.
Browser Isolation was a threat protection tool first — applied here to data protection. A common Zscaler pattern: reuse one capability across multiple domains.
