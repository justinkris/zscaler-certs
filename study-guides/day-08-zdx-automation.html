HALF A — ZDX OPERATIONS

Operating & Controlling ZDX

In Day 07, we built ZDX from the ground up — metrics, probes, scoring, alerts, and diagnostics. Today's first half focuses on who can see what (RBAC), what's happening on endpoints (Device & Process Inventory), and how the data is presented (Dashboards & Hosted Monitoring).

SUBTOPIC 1

RBAC in ZDX

Role-Based Access Control in ZDX is built on three interlocking layers. Understanding each layer individually — and knowing which symptom maps to which layer — is essential for the exam.

IMAGE PROMPT
A cartoon bouncer standing at a velvet rope outside an exclusive nightclub. The bouncer wears a sash that says "Role: Admin". He holds a clipboard and is only letting in guests from a specific list — the rope sign says "APAC Region Only" (Scope). Behind him, inside the club, a second smaller bouncer at an inner VIP door holds a sign: "Deep Trace — Special Access Required" (Permissions). A long queue of cartoon characters waits outside — some get waved through, some get turned away based on which badge they wear. Warm neon-lit nightclub atmosphere, slightly humorous cartoon style.

The Three Layers

Role
A named collection of permissions bundled together. Examples: Full Access, View Only, Custom. The role determines what actions an admin can perform in ZDX — but not on whom or what.
Scope
Restricts which users, devices, or locations the role applies to. Scope is configured using Locations, Departments, or Device Groups. A View-Only role scoped to APAC can only see APAC user data.
Permissions (within a Role)
Granular feature-level toggles embedded inside the role definition. Even within a broad role, specific high-sensitivity features (like Process Inventory, Deep Tracing, Packet Capture) must be explicitly enabled. Without these toggles, the button is greyed out — even if the admin has the correct role and scope.

ZDX Access Levels

Access LevelWhat it allowsTypical Use
FullView + configure everything within scopeZDX Administrators
View OnlyRead-only access to dashboards and dataHelp Desk, Auditors
CustomTailored combination of sections and permissionsScoped engineers, regional teams
NoneNo ZDX accessDefault for non-ZDX admins

Symptom → Cause Mapping

The exam describes an admin who can't do something — you need to identify which RBAC layer is the cause.

SymptomMost Likely Cause
Admin sees APAC users but not EU usersScope — restricted to APAC locations only
Admin can see all users but "Run Deep Trace" is greyed outPermission — granular Deep Trace permission not enabled in the role
Admin sees empty dashboards — no users, no devicesScope — assigned to a location/group with no users
Admin sees no menu items at allRole — no view permissions enabled in the role definition
!
WATCH OUT — EXAM TRAP
"Empty dashboards" could mean either Scope (location has no users) OR Role (no view permissions). Distinguish them: if the admin sees some dashboards but no data → likely Scope. If they see nothing at all, not even menu items → likely Role.
★
SENSITIVE FEATURES THAT NEED EXPLICIT PERMISSION
Process Inventory, Deep Tracing (Diagnostics), Packet Capture, and Bandwidth Tests all require their own granular permission toggle enabled within the role definition — on top of the base role. These expose detailed device and session data, so Zscaler gates them separately.
⟳
ANALOGY — The Hotel Keycard
Role = the type of keycard (guest, housekeeping, manager)
Scope = which floors and rooms the card opens
Permissions = whether you can just enter, or also adjust the thermostat and minibar settings

Device & Process Inventory

ZDX collects two types of endpoint inventory through the Zscaler Client Connector. Each answers a different troubleshooting question and has different timing characteristics.

IMAGE PROMPT
Two-panel cartoon at a mechanic's garage. Left panel: mechanic stands beside a shiny car, clipboard in hand, noting the make, model, licence plate, tyres. Label: "Device Inventory". Right panel: same mechanic has popped the hood — inside, tiny cartoon workers run around labelled "Chrome — 89% CPU", "Outlook — idle", "Antivirus scan". Mechanic peers in with wide eyes and a torch. Label: "Process Inventory". Same warm workshop lighting, playful cartoon style.
FeatureDevice Inventory 🚗Process Inventory 🔧
What it showsThe device itself — OS, version, hardware, hostname, ZCC version, locationWhat's running on the device — apps, CPU/memory usage per process
Question it answers"What is this machine?""What is this machine doing right now?"
Update frequencyPeriodic / session-basedMetrics updated every 1 minute
Display refresh—Top processes displayed every 5 minutes
PrerequisiteZCC installed + minimum versionZCC installed + minimum version + granular permission enabled
SensitivityModerateHigh — shows what apps/processes exist on a user's device
💡
MNEMONIC — Process Inventory Timing
"1 to update, 5 to show off"
The probe quietly updates metrics every 1 minute (collecting data continuously), but only refreshes the "top processes" display every 5 minutes. Like a chef tasting the soup every minute but only plating it for the food critic every 5. 🍲
!
WATCH OUT — ZCC IS ALWAYS THE PREREQUISITE
No ZCC = no Device Inventory, no Process Inventory, no probes, no ZDX score. The user simply disappears from ZDX entirely. Some advanced features also require a minimum ZCC version. If a question describes an admin who can see basic ZDX data but not advanced features, check for an outdated ZCC — it might not be an RBAC issue at all.

ZDX Dashboards & Hosted Monitoring

The Three Diagnostic Lenses

ZDX organises its data around three diagnostic viewpoints, each isolating a different potential layer of failure.

Left table (purple curtain): a sick-looking laptop lying on the table — doctor checking "pulse" with CPU bars and a health chart. Label: "Device". Centre table (blue curtain): a long winding road stretched out like a patient — doctor peering at individual "hops" with one section glowing red for packet loss. Label: "Network". Right table (green curtain): a web browser window lying flat — doctor tapping its "knee" with a reflex hammer, a clock measuring response time. Label: "Application". Playful, absurd medical cartoon style.
LensQuestion it answersKey metrics
📱 Device"Is the user's machine the problem?"CPU, memory, Wi-Fi signal, battery, ZDX device score
🌐 Network (Cloud Path)"Is the path between user and app the problem?"Hop count, latency, packet loss at each network hop
🖥️ Application"Is the app itself the problem?"Page Fetch Time, DNS Time, Server Response Time (TTFB), Availability

Additional Dashboard Views

  • User Dashboard — focused on a single user and their experience across all apps and devices
  • Geo / Locations View — where are problems concentrated geographically?
  • Department View — which org units are most affected?
  • Application Dashboard — all users of a specific app, showing average scores and trends over time
  • Incidents Overview — uses AI/ML to categorise and prioritise incidents across WiFi, Last Mile ISP, Zscaler Data Centre, and Application

Hosted Monitoring

Standard ZDX relies on ZCC running on real user endpoints. Hosted Monitoring runs probes from Zscaler-hosted synthetic endpoints in the cloud, completely independent of real users.

IMAGE PROMPT
Whimsical split illustration. Left side: a tired real office worker in a busy coffee shop — laptop open, weak Wi-Fi bars, spinning loading wheel, dead battery icon. Name badge reads "Real User (ZCC)". Right side: a cheerful shiny silver robot sitting completely alone in the same coffee shop at 3am — chairs stacked, lights off, wall clock shows 3:00 AM — the robot is perfectly content, sipping tea and tapping a glowing laptop. Robot badge says "Hosted Monitoring". Warm playful cartoon illustration style.
FeatureStandard ZDX (ZCC-based)Hosted Monitoring
Probe sourceReal user's device (via ZCC)Zscaler cloud-hosted synthetic endpoint
Requires real users?Yes — ZCC must be installed and runningNo — runs independently 24/7
Measures last-mile?Yes — full end-to-end from deviceNo — does not reflect user's last-mile experience
Best forReal user experience troubleshootingApp availability, pre-deployment baselining, lights-out sites
Works at 3am?Only if users are onlineYes — always running
★
HOSTED MONITORING USE CASES
1. 24/7 uptime monitoring — catch app outages at 3am when no users are online
2. Lights-out branch offices — monitor sites before or during deployment
3. Pre-deployment baselining — test a new SaaS app from multiple regions before rollout
4. Geographic coverage gaps — measure performance from regions with few real users
⟳
ANALOGY — Real User vs Hosted Monitoring
Standard ZDX = a mystery shopper who actually uses the store and reports the real experience.
Hosted Monitoring = a robot that periodically checks if the store's door opens and the lights are on — useful, but it can't tell you if the checkout is slow. 🤖🏪
💡
DECISION RULE — When to use which
Real users? → ZCC-based ZDX.
No users / synthetic / 24/7 / pre-deployment? → Hosted Monitoring.
HALF B — ZERO TRUST AUTOMATION

Making Zscaler Talk & Act

The second half of Day 08 shifts from observing your environment to automating action within it. Zero Trust Automation is about connecting Zscaler to the broader security ecosystem and enabling machine-speed responses — without waiting for a human to intervene.

Theme for Half B: Zero Trust only works at scale if it operates at machine speed. Humans can't review thousands of events per day. AI and automation close the gap — detecting, deciding, and acting without manual intervention.
SUBTOPIC 4

AI Influence & Workflow Automation

AI-Assisted vs AI-Automated

The most important distinction in this section — a frequent exam scenario — is the difference between Zscaler helping a human decide versus Zscaler acting on its own.

ModelWhat happensHuman in the loop?Response time
🤖 AI-AssistedDetects event, flags it, alerts admin, waits for human decision✅ YesMinutes to hours
⚡ AI-AutomatedDetects event, triggers action (block, quarantine, ticket) immediately❌ NoSeconds
→
EXAM SIGNAL PHRASES
"Automatically triggered" / "without manual intervention" → AI-Automated
"Notifies the administrator" / "recommends an action" → AI-Assisted

In a live compromise or ransomware scenario, AI-Automated is always the right answer — the time window matters critically. 30 minutes of human response time = a major breach.

Key AI Features in Zscaler

Zscaler AI Assistant 💬
A natural language interface built into the admin portal. Admins can ask "Why is this user's ZDX score low?" or "What policy is blocking this URL?" and receive AI-generated answers. Think: Google for your Zscaler deployment.
AI-Powered Policy Recommendations 📋
Analyses traffic patterns and suggests policy changes. Example: "47 users regularly access this app — consider adding it to your allowed list." The human still approves — this is AI-Assisted.
Smart Insights in ZDX 🔭
AI automatically surfaces the most likely root cause of a ZDX score drop. Instead of manually drilling through Device/Network/App dashboards, the system says "We think it's the Wi-Fi signal at the Sydney office." Automated first-level triage.

SUBTOPIC 5

OneAPI Framework

OneAPI is Zscaler's unified API gateway — a single standardised entry point for programmatic access to all Zscaler services (ZIA, ZPA, ZDX) through one endpoint, one authentication flow, and one token.

IMAGE PROMPT
Comedic side-by-side cartoon. Left panel "Before OneAPI": an overwhelmed developer sinking waist-deep in a flood of iron skeleton keys — keys overflowing from pockets, tangled in hair, hanging off both ears. Three separate desks each with a different angry login screen. Developer looks cartoonishly panicked. Right panel "After OneAPI": the same developer, impossibly calm, sitting back with ONE gleaming golden master key held up between two fingers. All three monitors show big green checkmarks. Developer wears a serene smile. Flat warm cartoon illustration style.

The Problem OneAPI Solves

Before OneAPI, any integration required separate registration, separate authentication, separate tokens, and separate endpoints for each Zscaler product. Three integrations = three times the complexity and three times the attack surface.

⟳
ANALOGY — Universal Power Outlet
Before international plug standards, every traveller needed a different adapter for every country. OneAPI is Zscaler agreeing on a universal plug standard — any tool that builds to the OneAPI spec can connect to any Zscaler product without custom wiring. 🔌🌍

OneAPI's Four Superpowers

SuperpowerWhat it meansExam relevance
🌍 Globally DistributedAPI gateways distributed across global regions for low latencyPerformance and availability of API calls
🗺️ Geographic AwareRequests automatically routed to the closest regional gatewayOptimises API performance without manual configuration
☁️ Zscaler Cloud AgnosticOne single endpoint for ZIA, ZPA, and ZDX — no juggling multiple URLsSimplifies integration code and reduces errors
🔑 Authenticate OnceZIdentity OAuth — one token grants access to all Zscaler servicesMost-tested feature — 3 tokens before → 1 token after
The core value proposition: Before OneAPI: 3 products = 3 registrations, 3 auth flows, 3 tokens, 3 endpoints. After OneAPI: 3 products = 1 registration, 1 auth flow, 1 token, 1 endpoint.

OneAPI Components

Access Token 🎫
Short-lived credential used to authenticate API requests. Deliberately expires quickly to minimise the window of opportunity if stolen. Used directly in all API calls.
Refresh Token 🔄
Longer-lived credential used only to obtain a new Access Token when it expires — without requiring the user to re-authenticate. Kept securely and never used directly in API calls.
API Gateway 🏢
The central hub that receives all OneAPI calls, enforces authentication, applies rate limiting, logs all interactions, and routes requests to the correct Zscaler service (ZIA/ZPA/ZDX).
⟳
ANALOGY — Access Token vs Refresh Token
Access Token = a daily visitor badge that expires at midnight. Used for everything during the day.
Refresh Token = the HR file locked in a drawer that issues new badges each morning. Rarely touched, kept safe, never worn directly.
★
WHY SHORT-LIVED ACCESS TOKENS?
Even if an access token is stolen or leaked — in logs, code, or a breach — it becomes useless the moment it expires. The attack window is dramatically limited. This is why the Access Token is short-lived and the Refresh Token (which can generate new ones) is kept separately and never included in API calls directly.
