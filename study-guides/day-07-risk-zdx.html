RISK MANAGEMENT

Risk360 — Holistic Risk Scoring

Risk360 solves a structural problem in enterprise security: every security tool produces its own risk score in its own format, with no common language between them. An attacker that hops across endpoint → identity → network → app → data is fragmented across multiple dashboards, and no single tool sees the full picture. Risk360 unifies all of this into one score, on one plane, with one language.

The one-liner: Risk360 gives you a single unified risk score pulled from Zscaler telemetry, config posture, and third-party signals — so you can answer "how risky are we right now?" with evidence, not gut feel.

Scoring Direction

!
WATCH OUT
High score = MORE risk = WORSE. Scale runs 0 (low risk) → 100 (critical). This is the opposite of ZDX scores. Dashboard colour-codes: green → yellow → orange → red.

The Four Risk Factors — Attack Chain Order

Factor Question it answers Attack chain stage Key Zscaler controls
Attack Surface "What can an attacker see and target?" Before the attack ZPA (inside-out), EASM, Tenant Restrictions
Compromise "How likely is initial foothold?" Getting in URL filtering, ATP, Sandbox, Browser Isolation
Lateral Propagation "Once inside, how far can they spread?" Moving around ZPA segmentation, Deception, ITDR Posture, CASB
Data Loss "What can they steal or exfiltrate?" Taking the prize Inline DLP, CASB API DLP, Endpoint DLP
⟳
ANALOGY
Fever thermometer, not credit score. A high reading means something is wrong. The score goes down as the organisation gets healthier — not up.

Data Sources

Risk360 pulls from three buckets: Zscaler inline telemetry (ZIA/ZPA logs, ZCC posture, ZDX scores), Zscaler config posture (how well your own tenant is configured — e.g., SSL inspection coverage), and third-party integrations (IdP, EDR, MDM, vulnerability scanners). Config posture is the often-overlooked bucket — Risk360 scores your own tenant hygiene, not just what's happening in traffic.

Outputs Beyond the Score

  • Remediation recommendations — ranked by score impact vs. effort, with deep-links into the admin console to fix them
  • Financial exposure quantification — estimated $ cost of a breach based on industry, size, and current risk factors. The board-ready number.
  • Board-ready reporting — one-click executive PDF: score, trend, top risks, top recommendations, financial exposure
PROACTIVE RISK TOOLS

UVM & Proactive Risk Capabilities

Unified Vulnerability Management (UVM)

CVSS tells you how bad a vulnerability theoretically is. UVM tells you what actually matters in your environment. It ingests vulnerability data from multiple scanners and enriches each finding with Zscaler-native context: exposure, reachability, exploitability, business impact, and compensating controls. The result is a single prioritised list of what to fix first — not a raw CVE dump.

Zscaler's unique edge: Because ZPA knows which apps are published and which users are entitled, UVM can compute actual attack paths to vulnerabilities — not just theoretical ones. A CVSS 9.8 on an app only reachable via ZPA is a very different risk than one exposed directly to the internet.
⟳
ANALOGY
UVM is the ER triage nurse. CVSS tells you who's bleeding; UVM tells you who to treat first based on severity, who's already stabilised (compensating controls), and who's about to walk out the door (active exploitation).

The Four Proactive Risk Capabilities

Capability Primary Risk360 Factor What it does Mental model
Deception Lateral Propagation Plants fake assets (decoy servers, credentials, files). Any interaction = confirmed attacker. Zero false positives. Silent tripwire in a dark room
ITDR Posture Lateral Propagation (identity) Continuously scans IdP (AD/Entra/Okta) for misconfigured identity attack paths — unconstrained delegation, shadow admins, stale accounts, priv-esc routes. BloodHound running 24/7 against your own directory
EASM Attack Surface Outside-in scan of the public internet for anything with your organisation's name on it — shadow IT, forgotten assets, exposed subdomains, leaked credentials. Googling your company as an attacker
Breach Predictor All four (forward-looking) ML models pattern-match your current telemetry against known attack progressions to forecast likely breach scenarios and predicted attack paths. Weather forecast for cyber attacks
★
MENTAL NOTE
Deception vs ITDR Posture both reduce Lateral Propagation — but via different vectors. Deception catches network/endpoint exploration (attacker moving between systems). ITDR Posture catches identity-based privilege escalation (attacker moving from normal user to admin via misconfigured AD). Two lenses on the same kill-chain stage.
!
WATCH OUT — ITDR POSTURE vs ITDR DETECTION
ITDR Posture = proactive — scans IdP config for weakness before any attack. ITDR Detection = reactive — behavioural monitoring that catches attacks in progress. If the question is about "misconfiguration" or "attack paths in AD" → Posture. If it's about "anomalous behaviour" or "real-time alerts" → Detection.
ZDX — DIGITAL EXPERIENCE

ZDX Scores — Device, Network, Application

ZDX solves the "user complains about app slowness" problem. In a traditional environment, IT has no visibility into the user's last-mile (home WiFi, ISP path) and no single source of truth for the full user-to-app journey. ZDX piggybacks on ZCC — already deployed for traffic forwarding — to continuously collect telemetry and probe business apps from the endpoint, with zero additional agent deployment.

!
SCORING DIRECTION — OPPOSITE OF RISK360
ZDX Score: HIGH = GOOD. LOW = BAD. A score of 90 = excellent experience. A score of 30 = terrible experience. Don't confuse with Risk360 (high = bad).

The Three Sub-Scores

Score Layer What it measures Low score means…
Device Score Endpoint CPU %, memory utilisation, disk IO, battery, OS version, top processes The laptop is the bottleneck — fix the machine
Network Score Path (WiFi → ISP → internet → Zscaler edge → app edge) Latency, jitter, packet loss, throughput — per hop via traceroute A specific hop on the path is degraded — drill in to find which
Application Score App / SaaS DNS resolution time, server response time, page fetch time, HTTP availability The app itself is slow — escalate to the SaaS vendor
The triage flow: Device Score low → fix the laptop. Network Score low → drill into hops (WiFi? ISP? Zscaler? App edge?). Application Score low → the SaaS is having a bad day, escalate with evidence.
⟳
ANALOGY
Fitness tracker for digital experience. ZCC is the watch already on the user's wrist. ZDX turns on the heart-rate sensor and GPS. The three sub-scores are like cardiovascular health (device), route quality (network), and destination quality (app).

The Four Classic Network Metrics

  • Latency — round-trip time (RTT). High latency = laggy feel, especially for real-time apps (Zoom, RDP)
  • Jitter — variation in latency between packets. High jitter = choppy voice/video even when average latency looks fine
  • Packet Loss — % of packets that don't arrive. Causes retransmissions → slowdowns, dropped audio, frozen video
  • Throughput / Bandwidth — data capacity of the path. Low throughput = slow file transfers and page loads
ZDX CONFIGURATION

Configuring Probes and Diagnostics

ZDX monitoring is built around three configuration decisions for each app you want to monitor: what (the Application target), how (the Probe type and frequency), and who (the monitored users or locations).

Probe Types

Probe Type Protocol What it measures Feeds
ICMP / Ping Probe ICMP Round-trip latency and packet loss to the target Network Score
Traceroute Probe ICMP / UDP Per-hop path mapping with latency and loss at each hop Network Score (per-hop)
Web Probe HTTP/HTTPS DNS resolution time, server response time, page fetch time, HTTP availability Application Score

Deep Trace — On-Demand Diagnostics

→
EXAM TIP — KNOW THIS DISTINCTION
Synthetic Probes = continuous background measurement → baseline, trend, alerting. Deep Trace = on-demand, admin-triggered, per-user + per-app → live troubleshooting when a user has an issue right now. If an exam scenario describes a user with a live active problem, the answer is Deep Trace — not "check the dashboard."
Synthetic Probes Deep Trace
When Continuous, scheduled On-demand, admin-triggered
Scope All monitored users Specific user + specific app
Purpose Baseline, trend, alerting Active troubleshooting of live issue
Output Score history charts Full per-hop diagnostic report at the moment of the issue
Analogy Fitness tracker running 24/7 Doctor running a full blood panel right now
