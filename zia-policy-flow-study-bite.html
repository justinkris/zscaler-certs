THE BIG PICTURE

Traffic classification is everything

Every packet that enters a ZIA Service Edge must be classified first. That single decision — is this web or non-web? — determines which inspection pipeline the traffic enters. Think of it like an airport: all travellers go through the same gate, but passengers split into domestic and international lanes before any detailed screening begins.

✈
MENTAL MODEL — THE AIRPORT FORK
Basic Firewall = the entry gate. Web traffic (HTTP/HTTPS) boards the Proxy plane. Non-web traffic (other protocols) boards the Advanced Firewall plane. They reunite at egress but take completely different journeys through the terminal.
The fork rule: HTTP/HTTPS → Proxy Engine (application layer, 6 stages). Everything else → Advanced Firewall (L3/L4 DPI, 5 stages). Same entry point, completely different pipelines.
WEB TRAFFIC PATH

HTTP/HTTPS — the Proxy Engine pipeline

Once Basic FW confirms a packet is HTTP or HTTPS, it enters the Proxy Engine — a deep inspection pipeline that operates at the application layer. Each stage feeds the next; a block at any stage ends the flow.

1
Traffic Classification — Basic FW Ingress gate. HTTP/HTTPS → Proxy lane. Other → Advanced FW lane.

EGRESS
★
EXAM NOTE
The canonical order to memorise: TLS Decrypt → URL Policy → ATP/DLP/Sandbox → DCC → FW/BWC. Questions often test where DLP sits relative to TLS decryption — answer: after. DLP requires decrypted content to inspect. If TLS inspection is bypassed for a destination, DLP cannot fire on it.

NON-WEB TRAFFIC PATH

Advanced Firewall — the DPI pipeline

Non-HTTP protocols (SSH, RDP, SMB, VoIP, FTP, custom TCP/UDP) skip the Proxy Engine entirely. They move directly to the Advanced Firewall (DPI) pipeline — deep packet inspection at Layer 3/4 with application awareness.

1
Traffic Classification — Basic FW Same entry gate. Non-HTTP → Advanced FW lane immediately.
INGRESS
2
Advanced Firewall (DPI) Top-down first-match rules · User/Group/Dept/Location · L4 port + L7 App ID · ALLOW / BLOCK / DROP / RESET
INSPECT
3
NAT Control DNAT (destination IP rewrite) · PAT (port rewrite) · Resolved before IPS so final routing is known
TRANSLATE
4
IPS — Intrusion Prevention System Signature + anomaly + heuristic · ThreatLabZ updates · ALLOW / BLOCK / DROP / BYPASS
PREVENT
5
Bandwidth Control (BWC) Guaranteed minimums for critical apps · Capped streaming · Time-based QoS rules
EGRESS

KEY DISTINCTION — FW POSITION
For web traffic, Firewall rules apply after proxy evaluation. For non-web, Firewall rules execute first. This is a common exam trap — the same FW engine, but different positions in the pipeline.
Non-Web Traffic — Advanced Firewall Pipeline
★
EXAM NOTE — NAT BEFORE IPS
NAT must be resolved before IPS — so that IPS evaluates the post-translation packet with correct final destination information. A question may ask why NAT precedes IPS: IPS needs to inspect the actual routed packet, not the pre-translation one.

RECAP

Five things to lock in

Study Bite Recap

01

Basic FW is the fork. Every packet hits it first. HTTP/HTTPS → Proxy Engine. Everything else → Advanced Firewall. Get this right and the rest of the flow falls into place.

02

Web order: TLS Decrypt → URL Policy → Security Stack (AV/ATP/DLP/Sandbox) → DCC → FW Policy → BWC. DLP only works because TLS is already decrypted by step 2.

03

Non-web order: Adv FW (DPI) → FW Policy → NAT → IPS → BWC. NAT comes before IPS so IPS sees the final translated packet.

04

The FW timing trap: For web, FW policy applies after the proxy. For non-web, FW policy applies first. Same engine, opposite position.

05

DCC closes the loop. After content inspection, AI/ML re-scores and feeds back into URL Policy — making enforcement dynamic, not just static rule-based.
